**Workcell هو منصة عمليات متعددة الوكلاء متخصصة في تشغيل مشاريع التطوير:
مجلس بشري يحدد الاتجاه، وفريق ذكاء اصطناعي — منسق، مطور،
مصمم، ضمان جودة — يُنجز المهام مع تقديم الأدلة.**

أنت تبقى في مقعد المجلس: تمتلك التوجيه والموافقات والسياسات. يضطلع الوكلاء بأدوار وظيفية،
ويستلمون المهام، ويتركون وراءهم نتائج عمل ملموسة و**أدلة** تُثبت إنجاز العمل فعلاً.
تدير لوحة التحكم المنظومة بأكملها — المشاريع والمهام والميزانيات والحوكمة
وسجل تدقيق ثابت لا يمكن تغييره — بينما تركز أنت على القرارات التي تهم.

> اعمل كشركة · نفّذ كمهام · اجعل التصميم مصدر الحقيقة · دع البشر يحكمون.

## الفلسفة

Workcell له رأي واضح حول طريقة إدارة مشاريع التطوير. أربعة التزامات ترسم ملامح
المنتج بأكمله:

### 1. الإنسان هو المجلس، لا متفرج

لا يوجد هنا "شركة بلا إنسان". الإنسان يمتلك التوجيه والموافقات والسياسات؛
والوكلاء يمتلكون التنفيذ. كل بوابة مهمة — الموافقة على التصميم، مراجعة الأدلة، الميزانية،
التوظيف — تنتهي عند قرار بشري مسجل في سجل تدقيق ثابت.

### 2. مشروع التطوير يُشحن بفريق حقيقي

يضم Workcell **أربعة مقاعد افتراضيًا — منسق، مصمم، مطور، ضمان جودة.** هذه
فلسفة متعمدة، لا مجرد قالب: هؤلاء الأربعة هم *أصغر* فريق قادر على أخذ فكرة من النية
إلى الإنجاز المثبت — التصميم أولاً، مع مالك واضح لكل بوابة.

| المقعد | الدور | المسؤولية |
| --- | --- | --- |
| **المنسق** | التوجيه والتنسيق | يحول اللغة الطبيعية إلى مهام منظمة، ويوجه العمل للدور الصحيح، ويراقب التشغيلات العالقة |
| **المصمم** | `designer` | نظام التصميم — يقترح مسودات التصميم (시안 — المخطط المرئي المُقدَّم)، ويحافظ على التصاميم المعتمدة كمصدر للحقيقة (**التصميم يأتي أولاً**) |
| **المطور** | `engineer` | التنفيذ والتصحيح والاختبارات — يبني استناداً إلى التصميم *المعتمد*، لا قبله |
| **ضمان الجودة** | `qa` | حكم *الإنجاز* — يُعيد الإنتاج والتحقق ويوقع على الأدلة |

يُهيئ الإعداد الأولي المنسق؛ وتعرض صفحة الوكلاء المقاعد الشاغرة كتوظيف بنقرة واحدة.
يوجّه ميثاق المنسق الكود نحو المهندسين، والتصميم البصري نحو المصممين، والتحقق نحو
ضمان الجودة — فشكل الفريق ليس مجرد توثيق، بل هو آلية تدفق العمل.

**المقاعد الأربعة هيكل، لا سقف — قم بالتوسع عليها بحرية.** وظّف أدواراً وظيفية إضافية حسب
متطلبات العمل — **قائد، مدير مشروع، باحث، كاتب، أمن، DevOps، أو وكيل للأغراض العامة** —
وزود أي وكيل بمهارات محددة النطاق ومكوّنات إضافية وخوادم MCP وأنظمة تصميم من سجل
القدرات. شغّل مالك مهمة كوكيل واحد أو — تجريبياً، قابل للتفعيل — كـ**دماغ مزدوج**
(نموذجان يولدان بالتوازي، ثم يدمجهما مُجمِّع). الوضع الافتراضي يحافظ على تماسك
مشروع جديد من اليوم الأول؛ ثم تنمو المنظومة لتناسب المشروع — لا العكس.

### 3. يُخطَّط للتطبيق بأكمله كمخطط واحد — التصميم هو مصدر الحقيقة

لكل مشروع **App Blueprint (전체 앱 기획 — مخطط التطبيق الكامل)**: عرض يرتكز على
التدفق بأسلوب Figma لشاشات التطبيق بأكملها، بحيث تعيش الخطة والتصميم في مكان واحد.

![App Blueprint — شاشات كتدفق، كل منها مقترنة بخطتها](docs/assets/app-blueprint.svg)

- **شاشة + خطة، كزوج.** كل شاشة عبارة عن **مسودة تصميم مرئية (시안 — المخطط المُقدَّم)** مقترنة
  بـ**خطة الشاشة (화면 기획 — screen plan)** — المواصفة للغرض والحالات والتفاعلات والبيانات.
  يُظهر المخطط *ما هي* الشاشة؛ والخطة تصفها. يُؤلَّفان ويتحركان معاً (شاشة واحدة = مسودة واحدة + خطة واحدة).
- **التدفق أولاً.** يفتح المخطط على التدفق: عُقد الشاشة مربوطة بسهام تنقل ذات تسميات،
  بحيث تكون تركيبة التطبيق بأكمله مقروءة دفعة واحدة. العُقد **قابلة للسحب مع حفظ المواضع**،
  القماش يتكبر عند المؤشر، والنقر على شاشة يفتح تفاصيل **خطة الشاشة (화면 기획)** — المخطط بجانب
  خطته، مع توضيح روابط الشاشة الواردة/الصادرة.
- **التصميم هو مصدر الحقيقة.** بالنسبة للعمل المتعلق بالشاشات، يتبع التنفيذ التصميم —
  لا العكس. تمر مسودة التصميم الأولية (시안) للمهمة بمراجعة بوابة (`needs_board_review →
  approved | changes_requested`)؛ حتى يوافق المجلس، **يُوقف الوكلاء التطوير**؛
  بعد الموافقة يُحقن التصميم كهدف التنفيذ. الفرق الجديدة **أولوية التصميم افتراضياً**
  (المهام غير البصرية تختار الخروج لكل مهمة مع تقديم سبب).
- يؤلف وكيل المصمم كل شاشة كمسودة تصميم مرئية **بالإضافة إلى** خطتها، ويمكن إعادة تأليف
  التصاميم الموروثة في نفس النموذج المقترن.

### 4. الإنجاز يعني الإثبات

استعارةً من منهجية issueflow، تحمل كل مهمة معايير القبول والأهداف المستثناة
وسطح الأدلة. **لا يمكن للمهمة الوصول إلى حالة *إنجاز* بدون حزمة أدلة**، ودور ضمان
الجودة يمتلك الحكم، وإنهاء مهمة يطلق دورة تعلم مركب
(قائمة تحقق تلقائية → تعبئة تلقائية اختيارية بنموذج اللغة → مهام متابعة). المعرفة تتراكم
بدلاً من التبخر.

## منشعب من Paperclip، مُعاد بناؤه لمشاريع التطوير

بدأ Workcell كتفرع من **Paperclip** (`paperclipai`، مرخص MIT) — لوحة تحكم مفتوحة
المصدر مبنية جيداً لتنسيق فرق وكلاء الذكاء الاصطناعي: مخططات تنظيمية، نبضات حياة،
ميزانيات، حوكمة، نظام تذاكر، سجل تدقيق ثابت، وعزل حقيقي متعدد الشركات. لوحة التحكم
تلك هندسة حقيقية وصلبة، ويحتفظ بها Workcell كأساس له. نحن ممتنون لها، وقد حُفظت حقوق
نشر Paperclip الأصلية وإشعار ترخيص MIT في [`NOTICE`](./NOTICE).

تفرعنا لأن **فلسفة منتجنا اختلفت** — لا لأن أي شيء في Paperclip كان خاطئاً لأهدافه الخاصة.
يُؤطر Paperclip نفسه حول *شركات بلا إنسان*: قوة عمل ذكاء اصطناعي مستقلة "توظفها"
في هيكل تنظيمي CEO/CTO وتتراجع إلى حد بعيد. يأخذ Workcell الموقف المعاكس بشأن
دور الإنسان ويضيّق الهدف من "تشغيل أي عمل" إلى **تشغيل مشاريع التطوير بشكل جيد**.
هذا الاختلاف عميق بما يكفي ليغير نموذج المجال وتجربة المستخدم وتعريف "الإنجاز":

- **استعارة شركة CEO ← نموذج مجلس + منسق + أدوار وظيفية.** الإنسان هو
  **المجلس**؛ الوكيل الأعلى هو **منسق** يوجه وينسق. الوكلاء هم أدوار وظيفية
  (منسق، قائد، مدير مشروع، مهندس، مصمم، باحث، كاتب، ضمان جودة، أمن، devops، عام)،
  لا ألقاب للإدارة العليا.
- **انضباط التنفيذ: التصميم أولاً + الأدلة كبوابة.** تتحكم الموافقة على التصميم في
  التنفيذ؛ والأدلة تتحكم في *الإنجاز*؛ وضمان الجودة يمتلك الحكم؛ والتعلم المركب يُغلق
  الحلقة. لا شيء من هذا موجود في Paperclip الأصلي — إنه التغيير السلوكي الأكثر أهمية في التفرع.
- **Open Design + Graphify، مُدمجان.** يدمج Workcell عمليات التصميم بأسلوب
  [Open Design](https://github.com/nexu-io/open-design) (مصنوعات التصميم، بوابات
  المراجعة، مكوّن إضافي للوحة التصميم) و**رسم بياني للمعرفة** يتغذى من منتج رسم بياني
  الكود **Graphify** — بحيث يتنقل الوكلاء بين المهام والكود والقرارات والتصاميم كفهرس
  متصل واحد بدلاً من إعادة اكتشاف المستودع في كل تشغيل.
- **أنظمة تنسيق جديدة كلياً.** **سجل القدرات** (المهارات / المكونات الإضافية / MCP
  / أنظمة التصميم بنطاق ورؤية ومستويات ثقة)، **تداول الدماغ المزدوج**
  (وكيل واحد يراجع نفسه عبر نموذجين)، **جسر MCP** الصادر، وطبقة حارس/استرداد
  تطوي التشغيلات المنتهية-لكنها-عالقة بدلاً من تقديم أوراق عمل.
- **إنتاجية متعددة المستأجرين / i18n.** عزل محكم للمستأجرين، تدقيق كامل لحذف متتالي،
  تدويل كامل، وضع مظلم افتراضياً.

Workcell تفرع مستقل وليس تابعاً لـ Paperclip ولا معتمداً منه.

## الميزات الرئيسية

- **لغة طبيعية ← مهمة.** صف ميزة على المجلس ويصيغ المنسق
  مهمة منظمة بمعايير القبول والأهداف المستثناة وسطح الأدلة.
- **بوابة التصميم.** تنتظر المهام المتعلقة بالشاشات حتى يوافق المجلس على تصميم
  مصدر الحقيقة؛ يصبح التصميم المعتمد هدف التنفيذ المحقون في تشغيلات الوكيل.
- **إنجاز مشروط بالأدلة + موافقة ضمان الجودة.** تصل المهام إلى *الإنجاز* فقط مع أدلة؛
  سياسة التنفيذ توجه أول "إنجاز" تلقائياً إلى مراجعة ضمان الجودة.
- **رسم بياني المعرفة + Graphify.** رسم بياني بمؤشرات فقط يشمل المهام والكود والقرارات
  والخطط؛ يستوعب `workcell code-graph` تصدير Graphify لإضافة بنية الكود إلى الرسم البياني.
- **App Blueprint (전체 앱 기획 — مخطط التطبيق الكامل).** عرض يرتكز على التدفق بأسلوب Figma
  لكل شاشة في التطبيق — مسودة تصميم مرئية (시안) مقترنة بخطة الشاشة (화면 기획)،
  عُقد قابلة للسحب مع حفظ المواضع، تكبير عند المؤشر، سهام تنقل ذات تسميات، والنقر للوصول
  إلى خطة كل شاشة. لكل مشروع؛ المسودة المعتمدة (시안) هي هدف التنفيذ. (مكوّن Open Design الإضافي
  لا يزال يعرض المصنوعات وفروق الإصدارات ومعاينات معزولة في صفحة `/design` مخصصة.)
- **تداول الدماغ المزدوج** *(تجريبي، قابل للتفعيل)*. وكيل واحد، نموذجان: يولد كلاهما
  مرشحاً بالتوازي، ثم دماغ مُجمِّع يدمجهما في الإجابة النهائية
  (بأسلوب OpenRouter-Fusion)؛ التشغيلات الحية محكومة بعلامة (مغلقة افتراضياً).
- **أحضر وكيلك.** محولات Claude وCodex المحلية (بالإضافة إلى HTTP/عملية) تحت مخطط تنظيمي واحد.
- **سجل القدرات.** مهارات ومكونات إضافية وخوادم MCP وأنظمة تصميم مُعيَّنة لنطاق الشركة أو لكل
  وكيل، مع مستويات ثقة وحالات رؤية وموافقة المجلس.
- **جسر MCP (داخل + خارج).** خادم MCP الوارد يعرض API الخاص بـ Workcell كأدوات؛
  عميل MCP الصادر يتيح لـ Workcell استدعاء خوادم جانبية خارجية (محكوم بالقدرات، نطاقه المستأجر).
- **التحكم بالتكاليف والحوكمة.** ميزانيات لكل وكيل مع توقف صارم، ومركز الاستخدام
  بشارات دقة `Exact / Synced / Estimated`، وبوابات موافقة المجلس، وسجل تدقيق ثابت
  لنطاق الشركة.
- **عزل متعدد الشركات وi18n.** نشر واحد، شركات معزولة بالكامل؛ واجهة مستخدم
  مُدوَّلة؛ وضع مظلم افتراضياً.

فهرس ميزات مفصل ومحدث دائماً (مع وسوم `[Paperclip]` / `[Changed]` / `[New]`)
موجود في [`docs/FEATURES.md`](./docs/FEATURES.md).

## تداول الدماغ المزدوج (تجريبي)

يمكن تشغيل مالك المهمة كـ**وكيل واحد بدماغين** — نموذجان مُهيَّئان باستقلالية —
مُدمَجان **بأسلوب OpenRouter-Fusion**. يـ**يولد** كلا الدماغين **إجابة مرشحة بالتوازي وباستقلالية**
(لا يرى أي منهما مسودة الآخر)؛ ثم يقوم **دماغ مُجمِّع**
(الدماغ A افتراضياً) بالتوفيق بينهما في إجابة نهائية أقوى — محتفظاً بما أصاب كل منهما،
متجاهلاً الباقي، حلاً للتعارضات. اختر نموذجين *مختلفين* وستضيف تنوع النماذج على القمة.

![تداول الدماغ المزدوج](docs/assets/dual-brain.svg)

لماذا ينجح: معظم الفائدة تأتي من **خطوة الدمج ذاتها**، لا من تنوع النماذج فحسب.
عندما قاست OpenRouter نهج **Fusion** على معيار **DRACO** لأبحاث Perplexity العميقة،
أدى إقران **Claude Opus 4.8 مع *نفسه*** كلجنة من نموذجين إلى رفع درجته من **58.8% إلى 65.5%**
— لأن مسارين حتى للنموذج ذاته يتباعدان، ومُجمِّع يُوفق بينهما يتفوق على لقطة واحدة.
([مقال](https://datasciencedojo.com/blog/openrouter-fusion-api/)، [OpenRouter](https://openrouter.ai/).)

**الحالة: قابل للتفعيل، مغلق افتراضياً.** محرك الدمج — توليد موازٍ + تجميع — مُنفَّذ
ومختبر، لكن تشغيله بنماذج *حقيقية* محكوم بعلامة
(`WORKCELL_PAIR_LIVE_LLM`، حتى لا ينفق التطوير/CI بالخطأ) ويعمل كتشغيل تداول
وكيل مخصص قابل للاستعلام. راجع [`docs/FEATURES.md`](./docs/FEATURES.md) للنطاق التفصيلي
لكل علامة.

## البنية (تخطيط الـ monorepo)

Workcell هو مساحة عمل pnpm (Node 20+، pnpm 9.15+):

| المسار | الحزمة | الدور |
| --- | --- | --- |
| `server/` | `@workcell/server` | Express REST API + خدمات التنسيق (نبضات الحياة، التشغيلات، بوابة التصميم، الحوكمة، التدقيق) |
| `ui/` | `@workcell/ui` | React + Vite واجهة المجلس (يخدمها API في وضع التطوير) |
| `cli/` | `workcell` | CLI / ثنائي `workcell` — إعداد، تكوين، code-graph، مزامنة سحابية |
| `packages/shared/` | `@workcell/shared` | أنواع مشتركة، ثوابت، مُحقِّقات، عقود مسارات API |
| `packages/db/` | `@workcell/db` | مخطط Drizzle، هجرات، عملاء قاعدة البيانات (Postgres مضمّن في التطوير) |
| `packages/adapters/` | — | محولات الوكيل (claude / codex / …) |
| `packages/adapter-utils/` | `@workcell/adapter-utils` | أدوات مساعدة مشتركة للمحولات (حقن MCP، رسم التكاليف) |
| `packages/mcp-server/` | `@workcell/mcp-server` | خادم MCP الوارد (Workcell API ← أدوات) |
| `packages/mcp-bridge/` | `@workcell/mcp-bridge` | عميل MCP الصادر (Workcell ← خوادم MCP خارجية) |
| `packages/plugins/` | — | نظام المكونات الإضافية، SDK، موفرو البيئة المعزولة، مكونات إضافية نموذجية (بما فيها لوحة Open Design) |

عملية Node واحدة تشغّل API وPostgreSQL مضمّن وتخزين ملفات محلي في
التطوير؛ في الإنتاج تُوجّهه إلى Postgres الخاص بك.

## البدء

المتطلبات: **Node.js 20+**، **pnpm 9.15+**.

```bash
pnpm install
pnpm dev          # API + UI في وضع المراقبة
```

يُنشأ قاعدة بيانات PostgreSQL مضمّنة تلقائياً في التطوير — اتركْ
`DATABASE_URL` غير مُعيَّن لاستخدامها. البرامج النصية الشائعة (من `package.json`):

```bash
pnpm dev          # تطوير كامل (API + UI، مراقبة)
pnpm dev:server   # الخادم فقط
pnpm typecheck    # فحص الأنواع على مستوى مساحة العمل
pnpm test         # تشغيل Vitest المستقر (لا يشغّل Playwright)
pnpm build        # بناء جميع الحزم
pnpm test:e2e     # مجموعة متصفح Playwright (قابلة للتفعيل)
pnpm db:generate  # إنشاء هجرة قاعدة بيانات
pnpm db:migrate   # تطبيق الهجرات
```

أول تشغيل: يُنشئ معالج الإعداد فريقك (أولوية التصميم افتراضياً)، ويُهيئ
**المنسق**، ويفتح مهمتك الأولى. ثم وظّف بقية الفريق الموصى به —
مهندس، مصمم، ضمان جودة — من صفحة الوكلاء (نقرة واحدة لكل مقعد شاغر).

راجع [`AGENTS.md`](./AGENTS.md) لسير عمل المساهمين وقواعد الهندسة.
