あなたはボードであり続けます。方向性、承認、ポリシーはあなたが所有します。エージェントは機能的な役割を担い、課題を引き受け、作業成果と**それが実際に完了したという証明**の両方を残します。コントロールプレーンが組織全体を管理します — プロジェクト、課題、予算、ガバナンス、そして改ざん不可能な監査証跡。あなたは本当に重要な意思決定に時間を使えます。

> 会社のように運営し · 課題として実行し · デザインを真実の源泉とし · 人間に判断させる。

## フィロソフィー

Workcell は開発プロジェクトの運営方法について明確な哲学を持っています。製品全体を形づくる4つのコミットメントがあります。

### 1. 人間はボードであり、傍観者ではない

ここに「人間ゼロの会社」はありません。人間が方向性、承認、ポリシーを所有し、エージェントが実行を担います。重要なすべてのゲート — デザイン承認、証明レビュー、予算、採用 — は、改ざん不可能な監査ログに記録された人間の意思決定で終わります。

### 2. 開発プロジェクトは本物のチームで出荷する

Workcell は**デフォルトで4つの席を持ちます — Orchestrator、Designer、Developer、QA。** これはテンプレートではなく意図的な哲学です。この4つは、アイデアを意図から実証済みへと導ける*最小*のチームです — デザインファーストで、すべてのゲートに明確なオーナーがいます。

| 席 | 役割 | 担当 |
| --- | --- | --- |
| **Orchestrator** | ルーティングとコーディネーション | 自然言語を構造化された課題に変換し、適切な役割に作業を振り分け、停滞した実行を監視する |
| **Designer** | `designer` | デザインシステム — シアン（レンダリングされたモックアップ/デザイン案）を提案し、承認済みの信頼できるデザインを維持する（**デザインが最初**） |
| **Developer** | `engineer` | 実装、デバッグ、テスト — *承認済み*デザインに基づいて構築し、決してその先行はしない |
| **QA** | `qa` | *完了*の判定 — 再現、検証、証明へのサインオフ |

オンボーディングでは Orchestrator がシードされます。Agents ページには不足している席がワンクリックで採用できる形で表示されます。Orchestrator のチャーターは、コードをエンジニアへ、UX をデザイナーへ、検証を QA へルーティングします — チームの形状は単なるドキュメントではなく、作業の流れそのものです。

**4つの席はスケルトンであり、上限ではありません — 自由に拡張できます。** 作業の要求に応じて追加の機能的役割を採用できます — **Lead、PM、Researcher、Writer、Security、DevOps、または汎用エージェント** — そして、Capability Registry からスコープ付きスキル、プラグイン、MCP サーバー、デザインシステムを任意のエージェントに装備できます。課題のオーナーを単一エージェントとして、または — 実験的でオプトイン — **デュアルブレイン**（2つのモデルが並列生成し、シンセサイザーがマージする）として実行できます。デフォルトは新しいプロジェクトを初日からコヒーレントに保ち、その後、プロジェクトに合わせて組織が成長します — 逆ではありません。

### 3. アプリ全体を1つのブループリントとして計画する — デザインが真実の源泉

すべてのプロジェクトには **App Blueprint（전체 앱 기획）**があります。これはアプリ全体の画面をフローファーストで Figma スタイルで表示したもので、計画とデザインが1つの場所に存在します。

![App Blueprint — フローとしての画面、それぞれに計画が対応](docs/assets/app-blueprint.svg)

- **画面とプランのペア。** 各画面は**純粋なシアン（レンダリングされたモックアップ）**と、その**화면 기획（画面プラン）** — 目的、状態、インタラクション、データの仕様 — がペアになっています。モックアップは画面が*何であるか*を示し、プランはそれを説明します。これらは一緒に作成され、一緒に移動します（1画面 = 1シアン + 1プラン）。
- **フローファースト。** ブループリントはフロー上で開きます。画面ノードがラベル付きナビゲーション矢印でつながり、アプリ全体の構成が一目でわかります。ノードは**ドラッグで位置変更でき、位置が保存されます**。キャンバスはカーソル位置でズームし、画面をクリックするとその**화면 기획**の詳細が開きます — プランの横にモックアップが表示され、その画面の受信/送信リンクも明示されます。
- **デザインが真実の源泉。** 画面に関わる作業では、実装はデザインに従います — 逆ではありません。課題の主要なシアンはレビューゲートを通過します（`needs_board_review → approved | changes_requested`）。ボードが承認するまで、エージェントは**開発を保留**します。承認後、デザインは実装ターゲットとして注入されます。新しいチームは**デフォルトでデザインファースト**です（非視覚的な課題は課題ごとに理由を添えてオプトアウトできます）。
- デザイナーエージェントは各画面を純粋なシアン**プラス**そのプランとして作成します。レガシーデザインも同じペアモデルに再作成できます。

### 4. 完了とは証明済みを意味する

issueflow の規律を取り入れ、すべての課題は受け入れ条件、非目標、証明サーフェスを持ちます。課題は**証明バンドルなしに*完了*に到達できず**、QA 役割が判定を所有し、課題の完了はコンパウンドラーニングサイクルを開始します（自動チェックリスト → オプションの LLM 自動入力 → フォローアップ課題）。知識は蒸発するのではなく、積み重なります。

## Paperclip からフォークし、開発プロジェクト向けに再構築

Workcell は **Paperclip**（`paperclipai`、MIT ライセンス）のフォークとして始まりました。Paperclip は AI エージェントチームのオーケストレーションのための優れたオープンソースコントロールプレーンです。組織図、ハートビート、予算、ガバナンス、チケットシステム、改ざん不可能な監査ログ、真のマルチカンパニー分離を備えています。そのコントロールプレーンは本物の堅固なエンジニアリングであり、Workcell はそれを基盤として維持しています。私たちはそれに感謝しており、Paperclip のオリジナルの著作権と MIT 許可通知は [`NOTICE`](./NOTICE) に保存されています。

フォークした理由は**製品哲学の相違**であり、Paperclip 自体の目標に何か問題があったからではありません。Paperclip は*ゼロヒューマン企業*を中心に据えています。CEO/CTO の組織図に「採用」し、ほぼ手放しで動く自律的な AI 労働力です。Workcell は人間の役割について正反対のスタンスをとり、「あらゆるビジネスを運営する」から**開発プロジェクトをうまく運営する**へと目標を絞り込みました。この違いはドメインモデル、UX、「完了」の定義を変えるほど深いものです。

- **CEO-会社メタファー → ボード + オーケストレーター + 機能的役割モデル。** 人間は**ボード**、トップエージェントはルーティングとコーディネーションを行う**Orchestrator**です。エージェントは機能的役割（orchestrator、lead、PM、engineer、designer、researcher、writer、QA、security、devops、general）であり、C スイートの肩書きではありません。
- **デザインファースト + 証明ゲート実行規律。** デザイン承認が実装をゲートし、証明が*完了*をゲートし、QA が判定を所有し、コンパウンドラーニングがループを閉じます。これらはストックの Paperclip には存在せず、フォークの最も重要な行動変更です。
- **Open Design + Graphify の統合。** Workcell は [Open Design](https://github.com/nexu-io/open-design) スタイルのデザイン操作（デザインアーティファクト、レビューゲート、デザインダッシュボードプラグイン）と、**Graphify** コードグラフプロデューサーが供給する**ナレッジグラフ**を統合します。エージェントは課題、コード、意思決定、デザインを、毎回リポジトリを再探索するのではなく、1つの接続されたインデックスとしてナビゲートできます。
- **ネット新規オーケストレーションサブシステム。** **Capability Registry**（スコープ、可視性、信頼ティアを持つスキル/プラグイン/MCP/デザインシステム）、**デュアルブレイン熟考**（1つのエージェントが2つのモデルで自己レビュー）、アウトバウンド **MCP ブリッジ**、完了したが停滞した実行をペーパーワークなしに折りたたむウォッチドッグ/リカバリーレイヤー。
- **マルチテナント / i18n プロダクト化。** 強化されたテナント分離、完全な削除カスケード監査、ファーストクラスの国際化、デフォルトのダークテーマ。

Workcell は独立したフォークであり、Paperclip とは提携しておらず、Paperclip の推薦も受けていません。

## 主な機能

- **自然言語 → 課題。** ボードで機能を説明すると、Orchestrator が受け入れ条件、非目標、証明サーフェスを持つ構造化された課題を下書きします。
- **デザインゲート。** 画面に関わる課題は、ボードが信頼できるデザインを承認するまで保留されます。承認されたデザインはエージェント実行に注入される実装ターゲットになります。
- **証明ゲート完了 + QA サインオフ。** 課題は証明エビデンスがある場合のみ*完了*に到達します。実行ポリシーが最初の「完了」を自動的に QA レビューにルーティングします。
- **ナレッジグラフ + Graphify。** 課題、コード、意思決定、計画をまたぐポインタのみのグラフ。`workcell code-graph` が Graphify エクスポートを取り込み、コード構造がグラフに加わります。
- **App Blueprint（전체 앱 기획）。** アプリのすべての画面をフローファーストで Figma スタイルで表示 — 純粋なシアンと화면 기획（画面プラン）のペア、ドラッグ可能な永続化ノード、カーソルズーム、ラベル付きナビゲーション矢印、各画面のプランへのクリックスルー。プロジェクトごとに設定され、承認されたシアンが実装ターゲットになります。（Open Design プラグインは引き続き専用の `/design` ページでアーティファクト、バージョン差分、サンドボックスプレビューをレンダリングします。）
- **デュアルブレイン熟考** *（実験的、オプトイン）*。1つのエージェント、2つのモデル。両方が並列で候補を生成し、シンセサイザーブレインがそれらを最終回答にマージします（OpenRouter-Fusion スタイル）。ライブ実行はフラグでゲートされています（デフォルトはオフ）。
- **独自エージェントの持ち込み。** Claude と Codex のローカルアダプター（HTTP/プロセスも含む）を1つの組織図のもとで利用できます。
- **Capability Registry。** スキル、プラグイン、MCP サーバー、デザインシステムを会社またはエージェントごとのスコープで割り当て。信頼ティア、可視性状態、ボード承認を管理します。
- **MCP ブリッジ（入力 + 出力）。** インバウンド MCP サーバーが Workcell の API をツールとして公開し、アウトバウンド MCP クライアントが Workcell から外部サイドカーを呼び出せるようにします（ケイパビリティゲート、テナントスコープ）。
- **コスト管理とガバナンス。** エージェントごとのハードストップ付き予算、`Exact / Synced / Estimated` 精度バッジ付きの Usage Center、ボード承認ゲート、改ざん不可能な会社スコープの監査ログ。
- **マルチカンパニー分離 & i18n。** 1つのデプロイで多数の完全分離された会社を運営。ユーザー向け UI は国際化済み。デフォルトはダークテーマ。

`[Paperclip]` / `[Changed]` / `[New]` タグ付きの詳細で常に最新の機能インベントリは [`docs/FEATURES.md`](./docs/FEATURES.md) にあります。

## デュアルブレイン熟考（実験的）

課題のオーナーは **2つのブレインを持つ1つのエージェント** として実行できます — 2つの独立して設定されたモデルが **OpenRouter-Fusion スタイル**で融合されます。両方のブレインが**並列かつ独立して候補回答を生成**し（どちらも相手の下書きを見ない）、その後**シンセサイザーブレイン**（デフォルトはブレイン A）が2つを1つの強い最終回答に統合します — それぞれが正しく得たものを保持し、残りを削除し、競合を解消します。2つの*異なる*モデルを選ぶと、合成の上にモデルの多様性が重なります。

![デュアルブレイン熟考](docs/assets/dual-brain.svg)

なぜ機能するか: 効果のほとんどは、モデルの多様性だけでなく**合成ステップ自体**から来ています。OpenRouter が Perplexity の **DRACO** 深層リサーチベンチマークで **Fusion** アプローチを測定した際、**Claude Opus 4.8 を*それ自体*と**2モデルパネルとしてペアにしただけで、スコアが **58.8% から 65.5%** に向上しました — 同じモデルでも2回通すと発散し、それらを統合するシンセサイザーがシングルショットを上回るためです。
（[解説記事](https://datasciencedojo.com/blog/openrouter-fusion-api/)、[OpenRouter](https://openrouter.ai/)）

**ステータス: オプトイン、デフォルトはオフ。** 融合エンジン — 並列生成 + 合成 — は実装されテスト済みですが、*実際の*モデルで駆動することはフラグ（`WORKCELL_PAIR_LIVE_LLM`、開発/CI が誤って費用を使わないよう）でゲートされており、専用のポーリング可能なエージェント熟考実行として動作します。正確なフラグごとのスコープについては [`docs/FEATURES.md`](./docs/FEATURES.md) を参照してください。

## アーキテクチャ（モノレポレイアウト）

Workcell は pnpm ワークスペース（Node 20+、pnpm 9.15+）です。

| パス | パッケージ | 役割 |
| --- | --- | --- |
| `server/` | `@workcell/server` | Express REST API + オーケストレーションサービス（ハートビート、実行、デザインゲート、ガバナンス、監査） |
| `ui/` | `@workcell/ui` | React + Vite ボード UI（開発時は API が配信） |
| `cli/` | `workcell` | CLI / `workcell` バイナリ — オンボード、設定、code-graph、クラウド同期 |
| `packages/shared/` | `@workcell/shared` | 共有型、定数、バリデーター、API パスコントラクト |
| `packages/db/` | `@workcell/db` | Drizzle スキーマ、マイグレーション、DB クライアント（開発時は組み込み Postgres） |
| `packages/adapters/` | — | エージェントアダプター（claude / codex / …） |
| `packages/adapter-utils/` | `@workcell/adapter-utils` | 共有アダプターユーティリティ（MCP インジェクション、コストマッピング） |
| `packages/mcp-server/` | `@workcell/mcp-server` | インバウンド MCP サーバー（Workcell API → ツール） |
| `packages/mcp-bridge/` | `@workcell/mcp-bridge` | アウトバウンド MCP クライアント（Workcell → 外部 MCP サイドカー） |
| `packages/plugins/` | — | プラグインシステム、SDK、サンドボックスプロバイダー、サンプルプラグイン（Open Design ダッシュボードを含む） |

開発時は1つの Node プロセスが API、組み込み PostgreSQL、ローカルファイルストレージを実行します。本番環境では独自の Postgres を指定します。

## はじめ方

要件: **Node.js 20+**、**pnpm 9.15+**。

```bash
pnpm install
pnpm dev          # API + UI をウォッチモードで起動
```

開発環境では組み込み PostgreSQL データベースが自動的に作成されます — `DATABASE_URL` を未設定のままにすると使用されます。共通スクリプト（`package.json` より）:

```bash
pnpm dev          # フル開発（API + UI、ウォッチ）
pnpm dev:server   # サーバーのみ
pnpm typecheck    # ワークスペース全体の型チェック
pnpm test         # 安定した Vitest 実行（Playwright は実行しない）
pnpm build        # すべてのパッケージをビルド
pnpm test:e2e     # Playwright ブラウザスイート（オプトイン）
pnpm db:generate  # DB マイグレーションを生成
pnpm db:migrate   # マイグレーションを適用
```

初回実行: オンボーディングウィザードがチームを作成し（デフォルトはデザインファースト）、**Orchestrator** をシードして最初の課題を開きます。その後、Agents ページから推奨チームの残り — Engineer、Designer、QA — を採用してください（不足している席ごとにワンクリック）。

コントリビューターのワークフローとエンジニアリングルールについては [`AGENTS.md`](./AGENTS.md) を参照してください。
