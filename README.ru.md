**Workcell — это платформа мультиагентных операций, специализированная для ведения проектов
разработки: совет директоров из людей задаёт направление, а команда ИИ — Orchestrator,
Developer, Designer, QA — доводит работу до результата с доказательством выполнения.**

[English](./README.md) · [한국어](./README.ko.md) · [日本語](./README.ja.md) · [简体中文](./README.zh-CN.md) · [繁體中文](./README.zh-TW.md) · [Español](./README.es.md) · [Français](./README.fr.md) · [Deutsch](./README.de.md) · [Português (BR)](./README.pt-BR.md) · [Русский](./README.ru.md) · [हिन्दी](./README.hi.md) · [العربية](./README.ar.md) · [Bahasa Indonesia](./README.id.md) · [Italiano](./README.it.md)

Вы остаётесь советом директоров: вы определяете направление, принимаете решения и устанавливаете
правила. Агенты берут на себя функциональные роли, подхватывают задачи и оставляют после себя
как результаты работы, так и **доказательства** того, что работа действительно выполнена.
Управляющий уровень ведёт организацию — проекты, задачи, бюджеты, управление и неизменяемый
журнал аудита — а вы тратите время на решения, которые действительно важны.

> Работайте как компания · исполняйте через задачи · проектируйте как источник истины · пусть люди судят.

## Философия

Workcell придерживается чётких взглядов на то, как должен быть устроен проект разработки.
Четыре принципа определяют весь продукт:

### 1. Человек — это совет директоров, а не наблюдатель

Здесь нет «компании без людей». Человек владеет направлением, согласованиями и политиками;
агенты владеют исполнением. Каждый ключевой контрольный пункт — согласование дизайна, проверка
доказательств, бюджет, найм — завершается решением человека, зафиксированным в неизменяемом
журнале аудита.

### 2. Проект разработки реализуется настоящей командой

Workcell по умолчанию включает **четыре роли — Orchestrator, Designer, Developer, QA.** Это
осознанная философия, а не шаблон: эти четыре роли — *минимально необходимая* команда, способная
провести идею от замысла до доказанного результата — с приоритетом дизайна и чётким владельцем
каждого контрольного пункта.

| Роль | Функция | Зона ответственности |
| --- | --- | --- |
| **Orchestrator** | маршрутизация и координация | превращает естественный язык в структурированные задачи, направляет работу нужным ролям и следит за зависшими процессами |
| **Designer** | `designer` | система дизайна — предлагает макеты-сианы (시안, отрендеренные черновики дизайна), поддерживает утверждённые эталонные дизайны (**дизайн — прежде всего**) |
| **Developer** | `engineer` | реализация, отладка, тесты — строит в соответствии с *утверждённым* дизайном, никогда не опережая его |
| **QA** | `qa` | вердикт *Готово* — воспроизводит, проверяет и подписывает доказательства выполнения |

При вводе в работу создаётся Orchestrator; страница Agents показывает незаполненные роли как
позиции для найма в один клик. Устав Orchestrator направляет код к инженерам, UX — к дизайнерам,
а верификацию — к QA, так что структура команды — это не просто документация, а способ движения
работы.

**Четыре роли — это скелет, а не потолок — свободно расширяйте поверх них.** Нанимайте
дополнительные функциональные роли по мере необходимости — **Lead, PM, Researcher, Writer,
Security, DevOps или агента общего назначения** — и оснащайте любого агента специализированными
навыками, плагинами, MCP-серверами и системами дизайна из Capability Registry. Запускайте
владельца задачи как одного агента или — в экспериментальном режиме по запросу — как **dual-brain**
(два отдельных мозга генерируют ответы параллельно, затем синтезатор объединяет их). По умолчанию
новый проект остаётся согласованным с первого дня; организация затем растёт под проект — а не
наоборот.

### 3. Всё приложение планируется единым чертежом — дизайн является источником истины

Каждый проект имеет **App Blueprint (전체 앱 기획, чертёж всего приложения)**: вид всех экранов
приложения в стиле Figma с акцентом на потоке, чтобы план и дизайн находились в одном месте.

![App Blueprint — экраны в виде потока, каждый с привязанным планом](docs/assets/app-blueprint.svg)

- **Экран и план — как единое целое.** Каждый экран представляет собой **чистый сиан (시안,
  отрендеренный макет)**, соединённый со своей **화면 기획 (планом экрана)** — спецификацией
  назначения, состояний, взаимодействий и данных. Макет показывает *что* такое экран; план
  описывает его. Они создаются и движутся вместе (один экран = один сиан + один план).
- **Поток — на первом месте.** Чертёж открывается на потоке: узлы-экраны, соединённые
  подписанными стрелками навигации, так что вся структура приложения читается с первого взгляда.
  Узлы **перетаскиваются с сохранением позиций**, холст масштабируется по курсору, а клик по
  экрану открывает его **화면 기획** в деталях — макет рядом с планом, с перечисленными входящими
  и исходящими ссылками этого экрана.
- **Дизайн — источник истины.** Для работы с экранами реализация следует за дизайном — никогда
  наоборот. Основной сиан задачи проходит контрольный пункт ревью (`needs_board_review →
  approved | changes_requested`); до согласования советом агенты **приостанавливают разработку**;
  после согласования дизайн внедряется как целевое задание для реализации. Новые команды работают
  **по принципу «дизайн первым» по умолчанию** (невизуальные задачи отказываются от этого
  принципа в индивидуальном порядке с указанием причины).
- Агент-дизайнер создаёт каждый экран как чистый сиан **плюс** его план, а унаследованные дизайны
  могут быть приведены к той же парной модели.

### 4. Готово — значит доказано

Следуя дисциплине issueflow, каждая задача содержит критерии приёмки, «не-цели» и поверхность
для доказательств. Задача **не может достичь статуса *Готово* без пакета доказательств**, роль QA
владеет вердиктом, а завершение задачи запускает цикл накопленного обучения (автоматический
чек-лист → опциональное автозаполнение LLM → дочерние задачи). Знания накапливаются вместо того,
чтобы улетучиваться.

## Форк Paperclip, переработанный для проектов разработки

Workcell начинался как форк **Paperclip** (`paperclipai`, лицензия MIT) — хорошо спроектированного
управляющего уровня с открытым исходным кодом для оркестрации команд ИИ-агентов: организационные
схемы, сигналы жизнеспособности, бюджеты, управление, система задач, неизменяемый журнал аудита и
настоящая изоляция мультикомпании. Этот управляющий уровень — настоящая, добротная инженерия, и
Workcell сохраняет её в качестве своей основы. Мы признательны за это, и оригинальное авторское
право Paperclip и уведомление о разрешении MIT сохранены в [`NOTICE`](./NOTICE).

Мы сделали форк, потому что наша **философия продукта разошлась** — а не потому, что что-то в
Paperclip было неверным для его собственных целей. Paperclip позиционирует себя вокруг *компаний
без людей*: автономная рабочая сила ИИ, которую вы «нанимаете» в организационную структуру
CEO/CTO и практически отходите в сторону. Workcell занимает противоположную позицию относительно
роли человека и сужает цель с «ведения любого бизнеса» до **эффективного ведения проектов
разработки**. Это различие достаточно глубоко, чтобы изменить доменную модель, UX и определение
«готово»:

- **Метафора CEO-компании → модель совета директоров + оркестратор + функциональные роли.**
  Человек — это **совет директоров**; главный агент — это **Orchestrator**, который маршрутизирует
  и координирует. Агенты — функциональные роли (orchestrator, lead, PM, engineer, designer,
  researcher, writer, QA, security, devops, general), а не должности C-suite.
- **Дисциплина выполнения «дизайн первым» + контроль через доказательства.** Согласование дизайна
  контролирует реализацию; доказательства контролируют *Готово*; QA владеет вердиктом; накопленное
  обучение замыкает петлю. Ничего этого нет в стандартном Paperclip — это наиболее значимое
  поведенческое изменение форка.
- **Open Design + Graphify, интегрированные внутрь.** Workcell интегрирует операции дизайна в стиле
  [Open Design](https://github.com/nexu-io/open-design) (артефакты дизайна, контрольные пункты
  ревью, плагин-панель дизайна) и **Knowledge Graph**, питаемый производителем кодового графа
  **Graphify** — чтобы агенты навигировали задачи, код, решения и дизайны как единый связанный
  индекс, а не обнаруживали репозиторий заново при каждом запуске.
- **Новые подсистемы оркестрации.** **Capability Registry** (навыки / плагины / MCP / системы
  дизайна с областью действия, видимостью и уровнями доверия), **dual-brain deliberation**
  (один агент с самопроверкой через две модели), исходящий **MCP bridge** и сторожевой уровень
  восстановления, который сворачивает завершённые, но зависшие процессы вместо составления отчётов.
- **Многопользовательская изоляция и i18n.** Укреплённая изоляция арендаторов, полные аудиты
  каскадного удаления, полноценная интернационализация, тёмная тема по умолчанию.

Workcell является независимым форком и не связан с Paperclip и не одобрен им.

## Ключевые возможности

- **Естественный язык → задача.** Опишите функцию на доске, и Orchestrator создаст структурированную
  задачу с критериями приёмки, «не-целями» и поверхностью для доказательств.
- **Контрольный пункт дизайна.** Задачи, связанные с экранами, приостанавливаются до тех пор, пока
  совет не согласует эталонный дизайн; согласованный дизайн становится целью реализации,
  внедряемой в запуски агентов.
- **Готово, контролируемое доказательствами + подпись QA.** Задачи достигают *Готово* только при
  наличии доказательств; политика исполнения автоматически направляет первый сигнал «готово» на
  проверку QA.
- **Knowledge Graph + Graphify.** Граф только из указателей поверх задач, кода, решений и планов;
  `workcell code-graph` принимает экспорт Graphify, чтобы структура кода вошла в граф.
- **App Blueprint (전체 앱 기획).** Вид каждого экрана приложения в стиле Figma с акцентом на
  потоке — чистый сиан в паре с 화면 기획 (планом экрана), перетаскиваемые узлы с сохранёнными
  позициями, масштабирование по курсору, подписанные стрелки навигации и переход к плану каждого
  экрана по клику. На уровне проекта; согласованный сиан является целью реализации. (Плагин Open
  Design по-прежнему отображает артефакты, различия версий и изолированные превью на выделенной
  странице `/design`.)
- **Dual-brain deliberation** *(экспериментально, по запросу)*. Один агент, две модели: оба
  генерируют кандидата параллельно, затем мозг-синтезатор объединяет их в финальный ответ
  (в стиле OpenRouter-Fusion); боевые запуски закрыты флагом (выключен по умолчанию).
- **Подключите своего агента.** Локальные адаптеры Claude и Codex (а также HTTP/process) в единой
  организационной структуре.
- **Capability Registry.** Навыки, плагины, MCP-серверы и системы дизайна, назначенные на уровне
  компании или для конкретного агента, с уровнями доверия, состояниями видимости и согласованием
  совета.
- **MCP bridge (входящий + исходящий).** Входящий MCP-сервер раскрывает API Workcell как
  инструменты; исходящий MCP-клиент позволяет Workcell обращаться к внешним сайдкарам
  (с контролем возможностей, в рамках арендатора).
- **Контроль затрат и управление.** Бюджеты на агента с жёсткими остановками, Центр использования
  с значками точности `Exact / Synced / Estimated`, контрольные пункты согласования совета и
  неизменяемый журнал аудита в рамках компании.
- **Изоляция мультикомпании и i18n.** Одно развёртывание, множество полностью изолированных
  компаний; UI интернационализирован; тёмная тема по умолчанию.

Подробный и актуальный реестр возможностей (с тегами `[Paperclip]` / `[Changed]` / `[New]`)
находится в [`docs/FEATURES.md`](./docs/FEATURES.md).

## Dual-brain deliberation (экспериментально)

Владелец задачи может быть запущен как **один агент с двумя мозгами** — двумя независимо
настроенными моделями — объединёнными **в стиле OpenRouter-Fusion**. Оба мозга **параллельно и
независимо генерируют ответ-кандидат** (ни один не видит черновик другого); затем **мозг-синтезатор**
(по умолчанию — мозг A) совмещает оба в один более сильный финальный ответ — сохраняя то, в чём
каждый оказался прав, отбрасывая остальное, разрешая конфликты. Выберите две *разные* модели — и вы
добавите разнообразие моделей поверх синтеза.

![Dual-brain deliberation](docs/assets/dual-brain.svg)

Почему это работает: большая часть эффекта обеспечивается **самим шагом синтеза**, а не только
разнообразием моделей. Когда OpenRouter измерял свой подход **Fusion** на глубоко-исследовательском
бенчмарке **DRACO** от Perplexity, использование **Claude Opus 4.8 с *самим собой*** в качестве
панели из двух моделей подняло его результат с **58.8% до 65.5%** — потому что два прохода даже
одной и той же модели расходятся, а синтезатор, который их примиряет, превосходит одиночный
запуск.
([статья](https://datasciencedojo.com/blog/openrouter-fusion-api/), [OpenRouter](https://openrouter.ai/).)

**Статус: по запросу, выключен по умолчанию.** Движок слияния — параллельная генерация + синтез —
реализован и протестирован, но запуск с *реальными* моделями закрыт флагом
(`WORKCELL_PAIR_LIVE_LLM`, чтобы dev/CI никогда не тратили деньги случайно) и выполняется как
выделенный, опрашиваемый запуск агентской делиберации. Точный охват по флагам приведён в
[`docs/FEATURES.md`](./docs/FEATURES.md).

## Архитектура (структура монорепозитория)

Workcell — это pnpm workspace (Node 20+, pnpm 9.15+):

| Путь | Пакет | Роль |
| --- | --- | --- |
| `server/` | `@workcell/server` | Express REST API + сервисы оркестрации (heartbeat, запуски, контроль дизайна, управление, аудит) |
| `ui/` | `@workcell/ui` | Board UI на React + Vite (в dev раздаётся API) |
| `cli/` | `workcell` | CLI / бинарник `workcell` — ввод в работу, настройка, code-graph, облачная синхронизация |
| `packages/shared/` | `@workcell/shared` | Общие типы, константы, валидаторы, контракты API-путей |
| `packages/db/` | `@workcell/db` | Схема Drizzle, миграции, клиенты БД (встроенный Postgres в dev) |
| `packages/adapters/` | — | Адаптеры агентов (claude / codex / …) |
| `packages/adapter-utils/` | `@workcell/adapter-utils` | Общие утилиты адаптеров (внедрение MCP, сопоставление затрат) |
| `packages/mcp-server/` | `@workcell/mcp-server` | Входящий MCP-сервер (API Workcell → инструменты) |
| `packages/mcp-bridge/` | `@workcell/mcp-bridge` | Исходящий MCP-клиент (Workcell → внешние MCP-сайдкары) |
| `packages/plugins/` | — | Система плагинов, SDK, провайдеры изолированной среды, примеры плагинов (включая панель Open Design) |

В development один Node-процесс запускает API, встроенный PostgreSQL и локальное хранилище файлов;
в production вы указываете на собственный Postgres.

## Быстрый старт

Требования: **Node.js 20+**, **pnpm 9.15+**.

```bash
pnpm install
pnpm dev          # API + UI в режиме наблюдения
```

В development встроенная база данных PostgreSQL создаётся автоматически — оставьте
`DATABASE_URL` не заданным, чтобы использовать её. Основные скрипты (из `package.json`):

```bash
pnpm dev          # полный dev (API + UI, watch)
pnpm dev:server   # только сервер
pnpm typecheck    # проверка типов по всему workspace
pnpm test         # стабильный запуск Vitest (без Playwright)
pnpm build        # сборка всех пакетов
pnpm test:e2e     # браузерный набор Playwright (по запросу)
pnpm db:generate  # генерация миграции БД
pnpm db:migrate   # применение миграций
```

При первом запуске мастер ввода в работу создаёт вашу команду (по умолчанию «дизайн первым»),
инициализирует **Orchestrator** и открывает вашу первую задачу. Затем наймите остальных
рекомендуемых участников команды — Engineer, Designer, QA — на странице Agents (один клик для
каждой незаполненной роли).

Рабочий процесс контрибьютора и правила инженерии описаны в [`AGENTS.md`](./AGENTS.md).
