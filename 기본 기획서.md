# 프로젝트 범용 멀티에이전트 운영 플랫폼 상세 기획서

## 제품 정의와 문서 우선순위

첨부된 기존 초안은 Paperclip의 control plane을 유지하면서 PairGroup, Usage Center, Capability Registry, Open Design 연동을 중심축으로 잡고 있었다. 이 문서는 그 초안을 바탕으로, 제품의 목적을 “무인 기업”이 아니라 **사용자가 큰 방향을 정하고, 에이전트들이 회사처럼 실행하는 프로젝트 운영 플랫폼**으로 재정의한 확장판이다. Paperclip는 스스로를 여러 AI 에이전트를 조직도·예산·거버넌스·작업·감사 로그로 운영하는 시스템으로 설명하고 있으며, issueflow-skills는 계획을 기능 목록이 아니라 사용자의 목표 경험과 이슈 단위 실행으로 연결하는 방식을 제시한다. 이 제품은 그 둘을 합쳐, **운영은 회사처럼, 실행은 이슈처럼, 판단은 인간이 하는 구조**를 목표로 한다.

이 문서는 구현 단계의 1차 source of truth다. 우선순위는 이 문서, 프로젝트별 계획 문서 세트, 이슈 명세, API/DB 계약, UI 상세 설계 순으로 둔다. 프로젝트별 계획 문서 세트(PlanSet)는 **세 종류의 계획 문서로만** 구성한다 — `SPEC`(전체 기획·source of truth), `implemented/`(구현된 것의 피쳐별 트리, 파일당 500줄 이하), `TODO`(앞으로 구현할 것 = SPEC − implemented). 상세 규약은 아래 「프로젝트 계획 문서 방법론」을 따른다. UI UX·QA 역할의 `DESIGN_DASHBOARD`·`QA_PLAYBOOK`은 별도 역할 산출물이며 같은 압축 규율을 따른다. issueflow-skills는 greenfield 프로젝트에서 plan anchor, goal experience summary, local backlog board, first issue wave, local proof entrypoint를 초기에 만들 것을 권장하고, active context 파일은 장기 기록 저장소가 아니라 현재 판단을 위한 얇은 포인터여야 한다고 본다. 이 문서는 그 철학을 플랫폼 레벨의 운영 규약으로 끌어올린다.

이 제품의 핵심 목적은 네 가지다. 첫째, 어떤 프로젝트든 같은 운영 모델로 다룰 수 있어야 한다. 둘째, 에이전트들은 채팅창이 아니라 **역할과 이슈**를 중심으로 움직여야 한다. 셋째, 사람이 없는 회사가 아니라 **사람이 방향과 승인권을 가진 회사**여야 한다. 넷째, 디자인·개발·검증·사용량·외부 능력 확장을 한 control plane 안에서 다뤄야 한다. Paperclip가 “보드가 승인하고, 비용을 통제하며, 조직도를 통해 작업을 위임하는 구조”를 강점으로 삼고 있고, issueflow-skills가 “사용자 경험–이슈–증거”를 연결하는 실행 철학을 강조한다는 점을 고려하면, 이 조합이 가장 자연스럽다.

## 레퍼런스 분석과 차별화

Paperclip에서 반드시 계승해야 할 것은 control plane이다. 공식 README 기준으로 Paperclip는 bring-your-own agent, org chart, heartbeat, budgets, governance, ticket system, immutable audit log, multi-company isolation, task-based persistence를 핵심 특징으로 제시한다. 또한 현재 저장소의 개발 지침은 single-assignee task model, atomic issue checkout, approval gates, budget hard-stop, activity logging을 핵심 불변식으로 명시한다. 따라서 이 제품은 Paperclip의 “조직 운영 계층”은 유지하되, pair work를 도입할 때도 **한 이슈의 상위 owner는 하나**라는 단순성은 지키는 편이 좋다. 즉, pair는 두 명의 독립 assignee가 아니라 **하나의 Work Owner 안에 들어 있는 실행 모드**로 설계해야 한다.

다만 Paperclip의 비전은 명시적으로 “zero-human companies”에 가깝다. 이 제품은 그 방향을 그대로 따르지 않는다. 여기서 사용자는 board이자 창업자이며, 에이전트는 자율 실행 주체이지만 최종 방향과 정책의 소유자는 사용자다. 따라서 거버넌스는 “사람을 제거하기 위한 승인 체계”가 아니라 “사람의 시간을 가장 가치 있는 의사결정에 집중시키기 위한 승인 체계”로 바뀌어야 한다. 이 차이 때문에 본 제품은 release sign-off, blocked issue 승인, skill/plugin/MCP 설치 승인, plan conflict 승인을 제품의 핵심 UX로 노출해야 한다. 또한 승인은 **두 모드**를 지원한다: 기본은 board(사용자)가 직접 결정하고, 사용자가 범위별로 **상시 위임(always approve)**을 켜면 그 범위의 승인을 **최상위 기획자 Agent가 사용자 대신 자동 처리**한다 — 저위험·반복 승인을 사람 루프에서 빼 "사람의 시간을 핵심 의사결정에 집중"시키는 장치다. 위임은 승인 type·프로젝트별 범위, 고위험 제외/한도, 자동 승인의 audit "위임에 의함" 표기, 즉시 취소를 갖춘다(상세 `DECISIONS.md` D18). Paperclip의 board/approval 개념은 그대로 유효하지만, 우리 제품의 철학은 자율성보다 **사용자 중심 실행 보조**에 더 가깝다.

issueflow-skills는 이 제품의 실행 철학에 더 직접적인 영향을 준다. 해당 저장소는 GitHub를 권장하지만 필수는 아니라고 밝히며, GitHub가 있을 경우 durable issue ID, PR linkage, review history, required checks를 협업·감사 계층으로 쓰고, 실제 실행은 여전히 저장소의 로컬 스크립트와 proof command를 기준으로 삼으라고 권한다. 또한 계획은 기능 나열이 아니라 사용자의 목표 경험을 설명해야 하며, 각 core issue는 최소 하나의 target user experience를 전진시켜야 하고, vertical slice를 통해 사용자 경로와 증거를 함께 묶어야 한다고 본다. 이 철학은 사용자가 말한 “칸반 TODO를 에이전트가 물고 가서 처리하는 구조”와 매우 잘 맞는다.

issueflow-skills의 또 다른 강점은 운영 중인 맥락을 작게 유지하는 습관이다. `PLAN_ANCHOR.md`와 `CURRENT_STATE.md`는 완료 이슈의 묘지나 무한한 메모장이 아니라 current macro direction, active issue, recent proof pointer, next action만 담는 얇은 파일이어야 하며, 대략 150~250줄을 넘거나 완료 이슈가 과도하게 쌓이면 compaction을 실행하라고 권장한다. blocked 상태의 자동화는 삭제하지 말고 `PAUSED`로 유지해 resume trail을 남기라고도 한다. 이 제품의 “기본 context를 주기적으로 compact하고, 필요 없는 skills는 평소에 감추고, 사용자 입력이 필요한 이슈를 보드에서 빠르게 확인한다”는 요구는 이 원칙을 거의 그대로 플랫폼 기능으로 만들면 된다.

Open Design은 디자이너 역할의 기준 구현체로 적합하다. 공식 README는 Open Design을 local-first, web-deployable, BYOK 구조의 오픈소스 디자인 엔진으로 설명하고, 기존 CLI 에이전트를 design engine으로 연결하며, sandboxed iframe preview, live todo progress, `<artifact>` 중심 생성 루프, 다수의 design system과 skills를 제공한다고 밝힌다. 더 나아가 read-only MCP server를 제공해 다른 에이전트가 Open Design 프로젝트를 직접 질의할 수 있도록 하고, plugin registry와 trust badge, headless-by-default, plugin-driven engine을 강조한다. 즉, Open Design은 단순히 “디자인 생성기”가 아니라, **디자인을 capability로 공급하는 서브시스템**이다. 본 제품에서는 이를 포크해서 본체에 녹이는 것보다, 기본 제공 capability pack과 sidecar bridge로 붙이는 편이 더 안전하다.

마지막으로, 사용자가 요구한 pair work는 실무 감각만이 아니라 연구 근거도 있다. Self-Refine는 same-model 생성–피드백–수정 루프가 one-shot 대비 평균적으로 더 나은 결과를 만들 수 있음을 보였고, Reflexion은 언어적 피드백과 episodic memory가 후속 시도 성능을 개선할 수 있음을 제시했다. Multiagent Debate는 다중 인스턴스의 토론이 factuality와 reasoning을 개선할 수 있다고 보고했고, sparse communication topology 연구는 모든 에이전트가 매번 전부와 대화하지 않아도 유사하거나 더 나은 성능을 더 낮은 계산비용으로 달성할 수 있음을 보였다. 따라서 본 제품의 pair loop는 “무한 토론”이 아니라 **최대 10회, 명시적 stop 조건, 비용 상한이 있는 2인 협업 루프**로 설계하는 편이 맞다.

## 프로젝트 범용 운영 모델

이 플랫폼은 소프트웨어 프로젝트뿐 아니라 콘텐츠, 디자인, 마케팅, 세일즈 운영, 고객지원 운영에도 적용할 수 있어야 한다. 이를 위해 프로젝트의 기본 단위는 “repo”가 아니라 **Project**로 두고, Project는 선택적으로 코드 저장소·이슈 트래커·디자인 작업공간·배포 대상·문서 저장소를 연결하는 구조로 설계한다. issueflow-skills가 말하듯 GitHub는 추천되는 협업·감사 계층일 뿐 필수는 아니며, 실행의 기준은 프로젝트가 가진 로컬 명령, 증거 수집 방식, 계획 문서다. 또한 goal-experience-planning은 운영 도구에도 target experience를 적용해야 한다고 말한다. 따라서 이 플랫폼의 단위는 특정 개발 환경이 아니라 **사용자가 결과를 얻고 싶은 업무 단위**가 된다.

신규 프로젝트의 bootstrap은 다음 순서로 고정하는 것이 좋다. 먼저 Planner Agent가 목표 사용자와 상황, 현재의 불편, 제품이 명확히 도움을 주는 순간, 이후 사용자가 할 수 있게 되는 것, first successful workflow, 비목표를 정리해 `PLAN.md`와 `PLAN_ANCHOR.md`를 만든다. 그 다음 stack decision, repo-topology decision, local backlog board, first numbered issue wave, local proof entrypoint를 만든다. 첫 wave는 가능한 한 얇은 end-to-end vertical slice를 포함해야 하고, 강한 사용자 경험 약속이 있다면 첫 화면부터 그 약속을 보여줘야 한다. issueflow-skills의 greenfield bootstrap, vertical-slice-architecture, experience-first-ui는 모두 같은 방향을 권한다.

기존 프로젝트를 가져오는 bootstrap은 조금 다르게 설계해야 한다. 먼저 Planner Agent는 repo, 이슈 트래커, 활성 PR, 현재 화면, 테스트, 문서, 디자인 패턴, backlog를 스캔해 현재의 사실 관계를 파악한다. 그 다음 `CURRENT_STATE.md`와 `BACKLOG_BOARD.md`를 생성하고, 열려 있는 작업을 `aligned`, `extension`, `conflict`, `deviation`으로 분류한다. ambiguous한 요구가 있으면 사용자가 모든 것을 다시 설명하게 하지 않고, 구현 방향을 바꾸는 질문만 한두 개 좁게 묻는다. issueflow-skills의 interactive brainstorming은 바로 이 절차를 권하고 있으며, GitHub adoption 문서는 GitHub를 audit layer로, 저장소의 로컬 명령을 execution layer로 취급하라고 분명히 말한다. 본 제품의 “기존에 관리되던 프로젝트를 이어서 이 플랫폼으로 관리하기 위한 bootstrap”은 이 철학을 UI/자동화로 제공하는 과정이 된다.

이슈 생성은 항상 자연어에서 시작할 수 있어야 한다. 사용자가 “이 기능을 처리해줘”, “이 화면을 더 명확하게 바꿔줘”, “이 흐름이 이상하다”라고 말하면 Planner Agent는 곧바로 새 issue를 draft로 만든다. ongoing process가 있어도, 독립적인 이슈라면 ready queue에 추가할 수 있어야 한다. 단, issue는 단순 메모가 아니라 acceptance criteria, non-goals, proof surface, 관련 계획 문서, 예상 owner role, blocker를 가져야 한다. 또한 broad request가 여러 독립 이슈를 숨기고 있으면 Planner Agent가 이를 분해해 TODO 여러 개로 만드는 것이 기본 동작이어야 한다. 이는 interactive brainstorming이 “큰 요청을 독립 이슈로 나누라”고 권하고, parallel delivery가 독립 lane의 병렬 처리를 권장하는 점과 일치한다.

칸반 상태 모델은 issueflow의 최소 보드 템플릿보다 더 풍부해야 하지만, 운영 맥락은 여전히 작게 유지해야 한다. 권장 기본 상태는 `Draft`, `Ready`, `In Progress`, `In Review`, `In QA`, `Blocked by User`, `Blocked by System`, `Done`, `Cancelled`, `Archived`다. 사용자는 `Done`, `Cancelled`, `Archived` 같은 상태를 숨길 수 있어야 하고, `Blocked by User`는 별도의 집중 레일로 보여줘야 한다. issueflow의 backlog template이 active context는 current pointers 중심으로 유지하고, 완료 이슈는 history로 이동하라고 요구하는 점을 고려하면, 칸반 UI도 “운영 중 상태”와 “기록용 상태”를 분리해서 다루는 것이 맞다. user-blocked automations를 삭제하지 말고 pause 상태로 남기라는 규칙도 그대로 적용해야 한다.

**완료 후 Compound(정리·학습) 단계.** 이슈/웨이브가 proof-gated Done에 도달하면, 메모를 따로 쌓는 대신 **하나의 정리 단계**를 거쳐 `Archived`로 넘어간다. 기획자 Agent가 주관하며 ① 완료분을 `implemented/`에 압축 반영하고 `TODO`에서 제거하며 지식 그래프(이슈↔구현↔결정 엣지·status)를 갱신하고, ② 작업 중 드러난 **재사용 학습·예방 규칙·실패한 접근**을 해당 `implemented/` 스냅샷의 짧은 제약/주의 노트나 지식 그래프 노드(QA는 `QA_PLAYBOOK`, 디자인은 `DESIGN_DASHBOARD`)에 **흡수**하고, ③ 자신이 건드린 범위를 훑어 발견된 debt·잠복 회귀·후속 슬라이스를 **후속 이슈로 환원**(없으면 그 근거를 남김)한다. 핵심 제약: 이 단계는 **새 문서 종류를 만들지 않는다** — 학습·예방은 D11의 세 문서(또는 역할 산출물)·그래프에만 녹인다. 이는 issueflow의 compound-learning(예방 규칙·실패 접근·후속 트리거·플랜 갭 캡처 + history compaction)을 제품 운영 모델의 1급 단계로 들여온 것이다. 문서 consolidation·후속 스윕·학습 흡수는 코어(P1~P2), 그래프 갱신 부분은 지식 그래프와 함께 P3.

## 프로젝트 계획 문서 방법론 (PlanSet)

Workcell이 관리하는 모든 프로젝트의 계획은 **세 종류의 문서로만** 표현한다. 완료/미완료 이슈 히스토리를 길게 누적해 계속 재독하는 방식은 채택하지 않는다. 프로젝트는 트리로 구성되어, 전부 읽지 않고 **필요한 파일만 골라** "앞으로 구현할 것(TODO)"과 "현재 구현된 스펙"을 비교할 수 있어야 한다.

- **SPEC — 전체 기획(source of truth).** 제품/프로젝트 전체 스펙의 단일 살아있는 진실원. 확정된 결정·아키텍처는 별도 결정 로그가 아니라 이 문서에 흡수한다.
- **implemented/ — 지금까지 구현된 것.** 단일 파일이 아니라 **큰 피쳐별로 나눈 폴더 트리**. 파일당 **500줄 이하**, 파일명이 내용을 드러낸다. 각 파일은 변경 이력이 아니라 **현재 구현된 상태의 압축 스냅샷**이며, 얇은 `INDEX.md`가 트리 맵 역할을 한다.
- **TODO — 앞으로 구현할 것.** = SPEC − implemented. 최상단에 현재 활성 이슈/다음 행동을 둔다. 미확정 결정은 "결정 대기"에 두고, 확정되면 즉시 SPEC에 반영한 뒤 제거한다.

**기획자 Agent**는 새 이슈를 만들고 완료시키는 흐름 속에서 `implemented/`와 `TODO`를 함께 유지보수한다. 파일/디렉터리명은 영어, 본문 카피는 한국어를 기본으로 한다.

### 불변 규칙 (HARD RULES — 위반 금지)

1. **이 세 종류(SPEC·implemented·TODO)가 계획·결정·이슈관리 문서의 전부다.** 그 외 메모·결정·상태·이슈관리용 문서를 새로 만들지 않는다. **명시된 이 방법 외의 방식으로 의견을 기억하거나 이슈를 관리하지 않는다.** (역할 산출물 `DESIGN_DASHBOARD`·`QA_PLAYBOOK`만 예외이며, 이들도 같은 압축 규율을 따른다.)
2. **이슈관리 문서는 어떤 경우에도 500줄을 넘지 않는다.** 압축이 최우선 — 줄일수록 좋다. 한도에 가까워지면 즉시 줄인다.
3. **더 이상 알 필요 없는 내용은 요약·아카이브하지 말고 삭제한다.** 완료/미완료 이슈 히스토리를 누적하지 않는다 — 완료된 것은 `implemented/`에 반영하고 `TODO`에서 제거한다.
4. **작은 요청으로 제품 스펙이 바뀌지 않으면 이 문서들을 건드리지 않는다.** 제품 스펙이 바뀔 때만 `SPEC`·`implemented/`·`TODO`를 함께 유지보수한다.

## 에이전트 체계와 페어워크

Paperclip의 예시 조직은 CEO, CTO, engineers, designers, marketers 같은 기성 역할 분할을 사용하지만, 이 제품은 **직함 중심 구조가 아니라 업무 흐름 중심 구조**를 채택한다. 상위 수준의 기본 역할은 `기획자 Agent`, `UI UX Agent`, `Developer Agent`, `QA / QC Auditor`로 고정하고, 그 외 도메인 특화 역할은 프로젝트 종류에 따라 추가한다. 즉, 사람 회사의 조직도를 흉내 내는 것이 목적이 아니라, 사용자가 실제로 맡기고 싶은 작업의 흐름을 가장 직접적으로 반영하는 구조를 만든다. Paperclip의 역할/조직도 강점은 유지하되, 역할명과 규정은 프로젝트 운영 철학에 맞춰 다시 써야 한다.

**기획자 Agent**는 사용자의 자연어 요청, 기존 계획 문서, backlog, 현재 화면, 활성 이슈, proof 상태를 종합해서 계획을 만들고 gap을 이슈로 환원하는 역할이다. 이 Agent는 다른 Agent가 일하고 있다는 이유로 멈추지 않는다. 독립적으로 처리 가능한 일이라면 새 issue를 계속 draft 또는 ready로 만든다. 또한 모든 계획 변경은 단순히 문장으로 끝내지 않고 **「프로젝트 계획 문서 방법론」에 따라 `SPEC`·`implemented/`·`TODO`와 acceptance criteria·non-goals·proof surface에 반영**한다. 이 Agent는 이슈를 만들고 완료시키며 `implemented/`(구현된 것)와 `TODO`(남은 것)를 늘 동기화하고, 문서가 500줄을 넘지 않도록 불필요한 내용을 삭제해 압축 상태를 유지한다. issueflow의 goal-experience-planning과 interactive brainstorming을 보면, 좋은 Planner는 기능 사양 작성자가 아니라 **사용자 경험을 issue graph로 번역하는 사람**이어야 한다. 또한 완료 후 **Compound 단계**(「범용 운영 모델」 참조)를 주관해 구현 반영·학습/예방 흡수·후속 이슈 환원을 책임진다.

**UI UX Agent**는 프로젝트마다 design dashboard를 source of truth로 유지하는 책임을 가진다. 화면이 추가되거나 바뀌면 항상 호출되어야 하고, deprecated된 화면은 기본 대시보드에서 제거되거나 archive로 이동되어야 한다. 이 Agent는 화면을 단순히 예쁘게 만드는 것이 아니라, 첫 뷰포트의 의미, 사용자 시선의 착지점, copy의 자연스러움, 버튼과 상태 메시지의 인간적인 톤, 정보 계층과 밀도를 검토해야 한다. issueflow의 experience-first-ui는 proof panel이나 dashboard-first UI를 경계하며, 화면이 테스트는 통과해도 조악하거나 어색하면 done이 아니라고 본다. Open Design은 이러한 역할을 수행할 수 있는 기본 디자인 capability와 preview infrastructure를 제공한다. 따라서 UI UX Agent에는 Open Design pack을 기본 탑재하고, 스크린 설계와 preview 생성은 가능한 한 이 경로를 우선 사용하게 해야 한다.

**Developer Agent**는 한 번에 한 feature issue를 맡는 것을 기본으로 한다. 단, 팀에는 여러 Developer Agent가 있을 수 있으며, 서로 독립적인 vertical slice라면 병렬 실행이 가능해야 한다. issueflow의 vertical-slice-architecture와 parallel-delivery는 issue 하나가 하나의 사용자 행동이나 결과를 담아야 하며, 가능한 한 worktree-first로 독립 lane을 나누고, root integration은 별도로 관리하라고 권한다. 따라서 Developer Agent는 “한 번에 한 issue” 규율을 지키되, 팀 레벨에서는 여러 developer lane을 운영할 수 있게 설계하는 것이 맞다. 특히 monorepo나 다중 패키지 환경에서는 package-local proof command와 disjoint ownership을 메타데이터로 명시해 병렬 충돌을 줄여야 한다.

**QA / QC Auditor**는 단순 테스트 실행기가 아니라, 완료 판정을 담당하는 독립 역할이다. 이 역할은 issue가 해결되었다는 **증거가 있을 때만** done 처리할 수 있고, 프로젝트별 QA 방법론을 스스로 축적해 `QA_PLAYBOOK.md`에 남겨야 한다. 웹 프로젝트에서는 Playwright 시나리오, 시각 요소가 중요한 앱에서는 golden test나 스냅샷 기준, copy 품질이나 화면 인상까지 포함한 qualitative audit를 함께 본다. issueflow는 proof를 real user path와 user-facing surface 중심으로 묶으라고 하고, experience-first-ui는 “green test인데도 화면이 못생기거나 카피가 어색하면 done이 아니다”라고 본다. 이 Agent는 바로 그 기준을 제품의 제도적 품질문지기로 구현한다.

그 외의 Agent는 프로젝트 특성에 따라 추가한다. 콘텐츠/마케팅에서는 Strategist, Researcher, Writer를 둘 수 있고, 세일즈/운영에서는 Prospector, Signal Analyst, Customer Support를 둘 수 있다. 다만 이들도 채팅형 assistant가 아니라 **issue owner 후보**이자 **proof를 남겨야 하는 역할**로 취급한다는 점은 동일하다. Paperclip의 강점이 다양한 agent runtime을 한 조직도 안에서 묶는 데 있는 만큼, 역할이 늘어나더라도 실행 규약은 같은 도메인 모델 위에서 유지하는 편이 맞다.

모든 Agent는 자연어로 성격을 바꿀 수 있어야 하며, 필요할 경우 system prompt를 직접 수정할 수도 있어야 한다. Anthropic의 subagents 문서는 subagent를 마크다운 파일과 YAML frontmatter로 정의하고, `/agents` 인터페이스에서 자연어 설명을 바탕으로 subagent를 생성하거나 편집할 수 있으며, body가 해당 subagent의 system prompt가 된다고 설명한다. 이 제품에서도 각 Agent는 `Role Charter`와 `Behavior Prompt` 두 층으로 분리해 관리하는 것이 좋다. 자연어 변경은 charter delta를 생성하고, 고급 사용자는 prompt editor에서 직접 body를 고칠 수 있게 하며, 모든 변경은 버전과 롤백 경로를 가져야 한다.

Pair work는 `WorkOwner = SingleAgent | PairGroup` 구조로 설계한다. PairGroup은 두 member와 하나의 역할 charter를 공유하되, 내부 실행은 라운드형 review loop로 돈다. 기본 루프는 A draft, B critique/edit, A critique/edit 순이며, 어느 한쪽이 `no_change_required`를 선언하면 종료한다. 최대 라운드는 10으로 고정하고, stop reason은 `converged`, `max_rounds`, `budget_stop`, `human_stop`, `blocked` 중 하나로 남긴다. 중요한 점은 pair가 더 많은 top-level assignee를 만드는 구조가 아니라, **한 issue owner 안에서 iterative refinement를 수행하는 실행기**라는 점이다. 이것이 Paperclip의 single-assignee 단순성을 해치지 않으면서 pair quality를 얻는 가장 안전한 방법이다.

```text
PairGroup 실행 기본 규약

1. 입력: issue spec, acceptance criteria, current context, latest artifact, proof expectations
2. Round 1: Member A가 초안 작성
3. Round 2: Member B가 요구사항 대비 결함/개선점 검토 후 수정안과 피드백 작성
4. Round 3+: 직전 수정본을 받은 반대편 member가 같은 방식으로 반복
5. 종료:
   - 한 member가 no_change_required 선언
   - maxRounds = 10 도달
   - budget/time guardrail 도달
   - user decision 필요로 blocked 전환
6. 산출:
   - 최종 artifact
   - round별 diff와 feedback summary
   - stop reason
   - 총 비용과 사용량
```

## 핵심 화면과 사용자 경험

제품의 기본 UI는 다섯 화면으로 요약할 수 있다. 첫째, **Project Home**은 현재 목표 경험, 활성 이슈, blocked 이슈, 최근 proof, 주요 비용 신호를 요약한다. 둘째, **Kanban Board**는 실제 운영 화면이다. 셋째, **Issue Workspace**는 한 이슈의 계획·산출물·pair log·proof·결정 요청을 보는 상세 화면이다. 넷째, **Design Dashboard**는 화면과 artifact preview의 source of truth다. 다섯째, **Capabilities / Usage Center**는 외부 능력과 비용을 관리한다. Paperclip가 이미 board, work, budgets, governance를 한 control plane에 모아 놓고 있고, Open Design이 preview와 live todo, comment-like annotation surface를 제공하므로, 이 다섯 화면만 잘 설계해도 제품의 핵심은 충분히 전달된다.

Kanban Board의 핵심은 “자연어 → 이슈” 변환과 “blocked decision”의 가시성이다. 보드 상단에는 자연어 입력창을 두고, 사용자가 “회원가입 흐름을 더 단순하게 바꿔줘”라고 쓰면 Planner Agent가 grounding 후 바로 draft issue를 만든다. 카드에는 title보다도 `owner role`, `single/pair`, `proof status`, `design impact`, `decision needed`, `usage burn`이 먼저 보여야 한다. 또한 사용자는 상태 필터에서 `Done`, `Cancelled`, `Archived`를 숨길 수 있어야 하고, `Blocked by User`는 오른쪽 고정 레일 또는 상단 우선 노출 섹션으로 분리하는 편이 좋다. issueflow의 active-context 원칙과 automation pause 규칙을 제품 화면으로 옮기면 바로 이런 UX가 된다.

Issue Workspace는 “왜 이 이슈가 존재하는가”와 “무엇으로 끝났는가”를 동시에 보여줘야 한다. 상단에는 issue classification, acceptance criteria, non-goals, plan link, owner, dependency, planned proof가 있고, 본문에는 artifact version history, diff, preview, annotations, run transcript summary, pair turns, audit events가 이어져야 한다. 완료 처리 버튼은 QA / QC Auditor 권한에서만 활성화하고, proof bundle이 없으면 disabled 상태를 유지하는 것이 좋다. 이는 issueflow가 모든 완료를 proof와 user-visible outcome에 묶으라고 권하는 것과 맞닿아 있다.

Design Dashboard는 UI UX Agent의 source of truth다. 여기에는 현재 유효한 화면 목록, deprecated 화면, 버전 비교, 승인 상태, linked issues, linked components, preview, annotations가 모두 있어야 한다. preview는 Open Design처럼 iframe 또는 artifact renderer를 통해 즉시 볼 수 있어야 하고, 사용자는 화면 위 특정 영역에 코멘트를 달 수 있어야 한다. Open Design은 sandboxed iframe preview, preview element comments, live artifact workflow, read-only MCP access를 제공하므로, 이 제품은 이를 참고해 **디자인은 문서가 아니라 미리보기 가능한 운영 자산**으로 다뤄야 한다. deprecated 화면은 기본 목록에서 숨기되 archive tab에서는 조회 가능하게 만드는 것이 적절하다.

Capabilities 화면은 회사가 보유한 skills, plugins, MCP servers, design systems를 하나의 registry로 보여주는 공간이다. 여기서 중요한 UX는 “모두를 보여주는 것”이 아니라 “필요한 것만 보이게 하는 것”이다. Anthropic의 skills 문서는 skill body가 실제로 호출될 때만 context에 로드되고, `skillOverrides`로 `on`, `name-only`, `user-invocable-only`, `off`를 제어할 수 있다고 설명한다. 따라서 본 제품의 capability registry도 `assigned`, `discoverable`, `hidden`, `disabled` 같은 visibility 상태를 가져야 하며, default는 role별로 필요한 능력만 보이게 하는 것이 맞다. Open Design의 plugin registry와 trust badge UX는 여기에 참고할 만한 훌륭한 선례다.

Usage Center는 사용자의 원래 요구사항을 가장 정직하게 반영해야 하는 화면이다. Anthropic는 Admin API에서 Usage & Cost API와 Rate Limits API를 제공하고, Claude Code/Claude 제품에서는 Settings > Usage와 billing surfaces를 통해 세션/주간 사용량, usage credits, spend/reporting을 보여준다. OpenAI는 API rate/usage limits와 response headers의 remaining requests/tokens, Usage page, Billing page, flexible pricing credits를 제공한다. 그러나 이 정보는 조직/워크스페이스 권한, 프로젝트 단위, 제품 플랜, rate-limit 종류에 따라 정확도와 갱신성이 다르다. 따라서 Usage Center는 `Exact`, `Synced`, `Estimated` 세 가지 정확도 배지를 가져야 하고, 사용자에게 “이 수치는 provider 공식 잔여량인지, 동기화된 관리 수치인지, 내부 burn-rate 추정치인지”를 명확히 알려야 한다.

## 시스템 아키텍처와 데이터 계약

권장 구현 전략은 **Paperclip 포크 + 도메인 재정의 + 외부 능력의 sidecar/bridge 통합**이다. 이유는 명확하다. Paperclip는 이미 Node.js server, React UI, task/work/governance/budget/activity/adapters/plugins 구조를 갖춘 “full control plane”이며, 우리가 원하는 회사형 orchestration의 기반을 상당 부분 제공한다. 반면 pair work는 현재 불변식 위에 얹는 수준을 넘어 execution owner 모델을 바꾸는 작업이고, 플러그인 쪽도 현재는 self-hosted, single-node, same-origin trusted UI라는 현실적 제약이 있다. 따라서 본 제품은 pair, design dashboard, capability governance를 core domain으로 편입하고, Open Design·MCP·외부 plugin은 선택적 연결 계층으로 다루는 편이 맞다.

아키텍처는 크게 `Control Plane API`, `Planner / Scheduler`, `Run Orchestrator`, `Adapter Gateway`, `Capability Registry`, `Design Sidecar Bridge`, `Usage Telemetry`, `Context Manager`, `Audit / Observability`로 나눈다. Adapter Gateway는 Claude Code, Codex, 기타 CLI 에이전트, HTTP/Webhook Agent, MCP Bridge를 수용하고, Design Sidecar Bridge는 Open Design과 같은 외부 artifact engine을 붙인다. Paperclip는 이미 Claude Code, Codex, CLI, HTTP 계열을 control plane 아래 놓는 구조를 설명하고 있고, Open Design은 headless-by-default와 read-only MCP를 제공하며, Anthropic subagents/skills는 독립 context와 tool scoping을 허용한다. 즉, 이 구조는 새로운 패러다임이라기보다 **검증된 요소를 목적에 맞게 재배치한 것**이다.

이 제품의 핵심 엔터티는 다음과 같이 고정하는 것이 좋다.

- `Company`: 조직, 정책, 구독, 예산, secret store, capability registry의 소유 단위
- `Project`: 특정 업무 단위. repo, tracker, design workspace, deployment target을 선택적으로 연결
- `PlanSet`: `SPEC`(전체 기획·source of truth) · `implemented/`(구현된 것, 피쳐별 500줄 이하 트리) · `TODO`(남은 것 = SPEC − implemented). 역할 산출물 `DESIGN_DASHBOARD`·`QA_PLAYBOOK`은 별도. 상세 = 「프로젝트 계획 문서 방법론」
- `Issue`: 목표 결과, acceptance criteria, proof surface, blockers, linked artifacts를 가진 실행 단위
- `WorkOwner`: `single` 또는 `pair`
- `AgentProfile`: 역할 charter, behavior prompt, model/runtime, permissions, memory policy
- `PairGroup`: 두 member, max rounds, stop policy, escalation policy
- `Run` / `PairTurn`: 실제 실행 로그, 비용, artifact diff, stop reason
- `Artifact` / `ArtifactVersion` / `Annotation`: 문서, 코드, 디자인, 리포트, preview 피드백
- `ProofBundle`: 테스트 결과, 스크린샷, golden, URL, QA verdict
- `Capability` / `CapabilitySource` / `CapabilityAssignment`: skill, plugin, MCP, design system의 등록과 배정
- `UsageSnapshot`: provider/plan별 사용량, 잔여량, 정확도, 갱신 시각
- `ContextShard` / `CompactionRun`: charter, working memory, episodic memory, capability index와 compaction 기록
- `DecisionRequest`: 사용자 승인, 정책 결정, 제품 방향 선택이 필요한 blocked 객체

이 엔터티 모델의 핵심은 `Issue`의 assignee를 여전히 하나로 유지하되, 그 하나의 owner가 single 또는 pair일 수 있게 하는 데 있다. 이것이 Paperclip의 single-assignee 단순성을 유지하면서, 요구된 수평적 pair work를 가장 안정적으로 수용하는 방법이다.

컨텍스트 관리는 네 개의 plane으로 분리하는 것이 좋다. `Role Charter`는 역할의 정체성과 금지 규칙, `Working Memory`는 현재 이슈와 blocker, `Episodic Memory`는 반복 교훈, `Capability Index`는 능력의 이름과 설명만 담는다. Anthropic skills 문서는 skill body가 필요할 때만 로드되며, `skillOverrides`를 통해 name-only 또는 off 상태를 둘 수 있다고 설명한다. issueflow는 active file budget을 작게 유지하라고 요구하고, hooks는 PreCompact/PostCompact를 통해 compaction 전후 반응을 걸 수 있게 한다. 따라서 이 제품은 full skill body, archive history, 과거 proof를 기본 preload하지 않고, 먼저 index와 pointer만 싣고 필요시 on-demand fetch하는 구조를 기본으로 삼아야 한다.

외부 능력 확장 계층은 세 가지로 나누는 것이 좋다. 첫째, **trusted platform modules**는 host와 밀접하게 통합되는 low-level adapter나 storage, secrets 확장이다. 둘째, **plugins**는 out-of-process worker를 가진 instance-wide additive capability다. 셋째, **skills / MCP / design systems**는 더 가벼운 prompt+tool 또는 data-source 확장이다. Paperclip의 plugin spec은 platform module과 plugin을 분리하고, third-party plugin을 out-of-process default로 보는 반면, 현재 구현의 plugin UI는 same-origin trusted code라고 경고한다. 따라서 본 제품은 third-party UI를 host에 직접 same-origin으로 마운트하지 말고, iframe/webview/typed bridge로 분리하는 쪽이 바람직하다. 동시에 Open Design의 read-only MCP처럼 “안전하지만 유용한” 브리지 패턴은 적극 활용할 수 있다.

업데이트 모델은 “자동 업데이트”가 아니라 **사용자 트리거형 staged update**를 기본으로 한다. Open Design은 plugin registry, trust badge, packaged auto-update를 제공하고, Paperclip의 장기 플러그인 설계는 hot install/uninstall/upgrade와 capability approval을 목표로 한다. 그러나 현재 Paperclip는 동적 플러그인 배포가 아직 cloud-ready가 아니며, same-origin UI 제약도 있다. 따라서 본 제품의 V1은 capability source를 import한 뒤 upstream repo/tag를 pinning하고, 사용자가 `check updates`를 눌렀을 때 manifest diff, license diff, smoke test 결과, breaking risk를 보여준 후 `apply update`를 승인하는 흐름이 가장 현실적이다. 회사 전체 반영과 특정 Agent만 반영하는 scope도 분리해야 한다.

## 프로젝트 지식 그래프 (코드·기획 인덱스)

계획 "문서"(SPEC/implemented/TODO)는 디테일을 담아 최소로 유지하고, 그 위에 **단일 경량 지식 그래프**를 둔다 — 코드 + 기획/스펙 + 결정 + 이슈(참조)를 노드/엣지로 잇는 **현재상태 지도/인덱스**다. 목적은 *기억*이 아니라 *탐색*: 많은 문서를 다 읽지 않고 그래프를 훑어 "어느 분야의 어떤 문서·코드·이슈를 자세히 볼지" 골라 on-demand 로드한다.

- **모델.** 노드=피쳐·모듈·개념·결정·계획항목·이슈(참조). 엣지=implements·depends-on·relates-to·spec-of·status. 각 노드는 디테일(문서 섹션·`file:line`·이슈 ID)으로의 **포인터**만 들고 본문을 복제하지 않는다(D11 정합).
- **채우기(파생) & 기존 자산 재사용.** 그래프는 권위 소스에서 **파생**되는 읽기 인덱스(불일치 시 소스 우선, 이벤트/온디맨드 갱신). 이미 있는 엣지를 **재사용**: `issue_relations`(blocks)·`issue_reference_mentions`(이슈↔이슈 자동 추출 링크)를 depends-on/relates-to로 흡수(병렬 저장 금지). 기존 `plugin-llm-wiki`의 **distillation 파이프라인**(이슈·댓글·문서를 페이지로 증류)을 그래프 **populator**로 재사용. 코드 노드는 Graphify식 코드 그래프화.
- **대체 대상 (기존 Paperclip 역할).** llm-wiki의 *내비게이션 레이어* — `index.md`(평면 카탈로그)+`[[wikilink]]` backlinks+`projects/standup.md` — 를 **정식 타입드 그래프가 승계**한다. wiki-maintainer 에이전트는 "위키 문서 관리"에서 "그래프 유지(populator)"로 재정의. 키워드 검색(`company-search`, trigram; 임베딩 없음)은 텍스트 매칭이라 **보완 공존**(그래프=구조 탐색, 검색=텍스트).
- **Agent가 언제 쓰나(touchpoints).** ① **MCP 그래프 도구**(`graphQuery`/`graphNeighbors`/`graphUpsert`) on-demand 질의·갱신 — 현재 MCP엔 검색/지식 도구가 없어 신설. ② **heartbeat-context**(이슈 wake 번들)에 그래프 파생 "관련 문서·결정·이슈 포인터" 주입 → 깰 때 관련 컨텍스트 자동. ③ 이슈 생성/완료 시 갱신. **기획자**=생성 전 현황 질의(중복·의존·영향), **Dev/QA**=관련 코드·문서·이슈 탐색.
- **충돌 없음.** 이슈 진실원은 DB/Kanban — 그래프는 ID 참조만(상태 재저장·이중 추적 금지). 시간축은 비핵심(불변 audit log가 "언제 바뀜" 담당 + D11 비누적과 충돌) → 필요 시 P3 옵션 레이어.

방식: **MCP 사이드카/브리지**(외부 능력 계층과 동일), 기본 preload 금지·on-demand. **백엔드 경량 우선** — 임베디드 Postgres 그래프 스키마/Apache AGE + pgvector(새 DB 없이), Neo4j/FalkorDB 미채택. 구현·백엔드 확정 = **P3 PoC**. (결정 = `DECISIONS.md` D12)

## Open Design 통합 (디자인 capability)

UI/UX Agent의 디자인 엔진으로 **Open Design**(`nexu-io/open-design`, Apache-2.0 — 로컬 데몬 + CLI 에이전트가 디자인 엔진, `<artifact>` 샌드박스 iframe 프리뷰, read-only MCP, BYOK, 150 design systems·132 skills)을 붙인다. 단 **코어 포크가 아니라 플러그인(capability pack) + 사이드카 브리지**로 — Open Design 엔진이 Workcell이 이미 가진 것(claude/codex 어댑터·스킬·워크스페이스·Run Orchestrator)과 크게 겹치기 때문.

- **이식 형태 = Workcell 플러그인**(`plugin-llm-wiki` 템플릿). 어댑터 아님 — 어댑터 계약(`ServerAdapterModule.execute`)은 "에이전트 프로세스 런타임"이고 OD는 엔진/MCP/아티팩트 소스라 부적합(빌트인 어댑터는 잠겨 있음). 플러그인이 제공: UX Agent **agent tools**, **디자인 Skills + `DESIGN.md` 시스템**(BYOK=`secrets.read-ref`), **Design Dashboard**(plugin `page` 슬롯, 라우트 `design`/`screens`; `design-guide`는 예약), 이슈별 아티팩트(`detailTab`/`taskDetailView`), **영역 주석 = 기존 `commentAnnotation` 슬롯 재사용**, `dashboardWidget`.
- **생성은 Workcell 어댑터/워크스페이스/Run Orchestrator로 실행**(OD 데몬을 제2 오케스트레이터로 붙이지 않음) — 디자인 스킬/시스템을 claude/codex 어댑터로 구동.
- **프리뷰 = 샌드박스 iframe** — 기존 iframe 런처(`/_plugins/:id/ui/*` + asset CSP `sandbox`) 재사용, in-process React 슬롯 로더는 금지(생성된 신뢰불가 HTML).
- **아티팩트 = 기존 `issue_work_products`(type/url/reviewState/isPrimary/metadata/createdByRunId) + `assets`(래스터/blob) 증강**. 스펙의 `Artifact/ArtifactVersion/Annotation`은 새 코어 테이블 대신 이 위에 매핑(주석=`commentAnnotation`+`issue_comments`). 병렬 시스템 금지.
- **외부 MCP 브리지 = 유일한 net-new 인프라**(Workcell엔 outbound MCP 클라이언트 없음). 두 경로 PoC: (a) 플러그인 worker `http.outbound`/신규 MCP 클라이언트, (b) claude/codex 어댑터에 OD read-only MCP 직접 부착(CLI가 MCP 네이티브 — 코어 변경 0 가능). **(b) 우선**, 그리고 이 브리지는 **지식 그래프(코드 그래프 MCP)와 공유** — 한 번 만들어 둘 다 쓴다.
- **UX Designer 역할은 이미 존재**(`uxdesigner.md`, role `designer`, "Visual-truth gate"=실제 뷰포트 렌더+스크린샷 의무) → Open Design이 그 게이트를 충족시키는 엔진을 공급. 기존 역할에 플러그인 도구만 장착. **대체가 아니라 증강**(디자인 기능 자체가 그린필드).
- **라이선스:** Apache-2.0 — 스킬/디자인시스템 *번들/포팅* 시 `NOTICE` 귀속 추가, *외부 사이드카 실행만* 하면 경량.
- **시퀀싱:** Design Dashboard 기본 + UX agent = P2, Open Design 브리지 + outbound MCP + capability registry = P3.

## 구현 그라운딩 (코드베이스 이식 지도)

> Paperclip 포크 위에서 무엇을 재사용하고 무엇이 net-new인지의 이식 지도(2026-05-27 코드베이스 조사). 새 도메인 1차 개념 — proof bundle · owner role · single/pair · design impact · decision-needed · usage burn · `Blocked by User` 레일 — 은 코드에 아직 없다(더 풍부한 상태모델도 `IssueStatus` 미반영).

**5개 화면 = 그린필드 제품 레이어 + 도너 페이지.** 현 UI는 Paperclip 범용 agent-ops UI(`docs/DESIGN_DASHBOARD.md`가 5화면 모두 미구현 명시). 진화 출발점:
- **Issue Workspace** ← `IssueDetail.tsx`(최강 도너: 트랜스크립트 `IssueChatThread`·런/비용 `IssueRunLedger`·버전 문서 `IssueDocumentsSection`·reviewers/approvers·첨부). net-new = **proof-gated Done**(QA 역할+proof bundle 조건)·classification/acceptance/non-goals/pair-turns/audit 타임라인.
- **Kanban** ← `KanbanBoard.tsx`+`IssuesList.tsx`(구조 有). net-new = **자연어→Planner 초안**(현 `POST .../issues`는 순수 CRUD·LLM 없음 = Phase 1 핵심 갭)·카드 칩row(owner role·single/pair·proof·design impact·decision·usage burn)·`Blocked by User` 전용 레일.
- **Project Home** ← `Dashboard.tsx`(전역)+`ProjectDetail.tsx`(overview) 목표경험 중심 재구성.
- **Design Dashboard · Capabilities/Usage Center** = 그린필드(도너: `DocumentDiffModal`·`CommentThread` / `CompanySkills`·`Costs`·`PluginManager`).

**PairGroup = 단단한 단일-run 기질 위 얇은 오케스트레이터.** 한 라운드 = 기존 `adapter.execute` 1회 = `heartbeat_runs` 1행(`executeRun`). 재사용: `agent_task_sessions`(멤버별 세션)·`heartbeat_run_events`+`TranscriptEntry.diff/.result`(라운드별 diff·피드백·비용)·`cost_events`/`agent_runtime_state`(비용)·wakeup→claim 경로. 이슈는 이미 단일 `assigneeAgentId`(=한 owner 불변식). net-new = ① 2-에이전트 교대 **라운드 루프 오케스트레이터**(stop: no_change_required/maxRounds=10/budget/human/blocked) ② **`PairTurn` 원장**(round#·drafter/critic·diff·feedback·cost·stop_reason) ③ **`WorkOwner` 인디렉션**(단일 assignee→pair 해소).

**Usage Center = 이미 ~70% 구현됨 (큰 발견).** 내부 burn(`cost_events`+`costService`)·예산(`budgetService`)에 더해 **provider 공식 quota sync가 이미 존재** — `getQuotaWindows()`가 claude-local(Anthropic OAuth usage + Claude CLI `/usage`)·codex-local(codex app-server RPC + ChatGPT WHAM)에서 가져옴(`/costs/quota-windows`, `Costs.tsx`·`ProviderQuotaCard`). → 배지 매핑: **`Synced/Exact`=구독/OAuth quota sync, `Estimated`=어댑터-보고 burn; metered API-key는 sync 불가라 구조적 `Estimated`.** net-new = 정식 `Exact|Synced|Estimated` provenance enum(현재 ad-hoc 라벨+`quotaSource` 문자열)·통합 Usage Center IA·(선택) `UsageSnapshot` 이력 저장(현재 quota는 live fetch·미저장). → 기존 `Costs.tsx`에서 진화.

**에이전트 charter/instructions.** 재사용: `agentInstructionsService` 파일 번들(에이전트별 `AGENTS.md`, managed|external 모드 = NL .md 편집)+`agent_config_revisions`(config 스냅샷·rollback). net-new = **charter(정체성·금지규칙)↔behavior(프롬프트 본문) 2층 분리** + **프롬프트 본문 버전/rollback**. ⚠️ 현 rollback은 **config 행만 복원** — 디스크의 `AGENTS.md` 본문은 스냅샷에 없음(본문 버전관리는 새로 구축).

**Capability registry/visibility.** 현재 3계통 분산(skills=`company_skills`, plugins=`plugins`/lifecycle, MCP/adapters=`registry`); 배정은 `adapterConfig.workcellSkillSync.desiredSkills`(에이전트별 skill key 배열)로 **암묵 저장**. net-new = **통합 registry(skill+plugin+MCP+design system) + 가시성 enum(assigned/discoverable/hidden/disabled) + 명시 scope 테이블**. evolve from `company-skills.ts`.

**보안 trust tier.** 재사용: 견고한 secret store(`company_secrets`/versions/`company_secret_bindings`)+감사(`secret_access_events`/`activity_log`)+플러그인 capability 승인 게이트(`plugin-lifecycle.ts`: 새 capability 추가 시 `upgrade_pending`→승인). net-new = **`trusted|reviewed|unreviewed` 등급 + 강제 번들**(auto-invoke 금지·기본 disabled·UI 격리·secret 차단). ⚠️ `company_skills.trustLevel`은 **"콘텐츠 위험 클래스"(markdown/scripts/assets)이지 보안 리뷰 상태가 아님** — 이름·의미 혼동 금지(새 필드로).

**Bootstrap (최대 그린필드).** 현재 수동 CRUD뿐: `onboard.ts`=인프라만, `createProject`/`createCompany`=행 삽입, `OnboardingWizard`=수동 4스텝(에이전트 1+이슈 1). company export/import는 마이그레이션용(≠bootstrap), repo 스캔은 skills에만. net-new = **Planner 주도 bootstrap**(신규=plan/anchor/backlog/proof+first wave / 기존=repo·tracker·test 스캔→current-state+분류 backlog). evolve hook=`createProject`/`OnboardingWizard`, 스캔 패턴=`company-skills` repo-scan.

**감사/버전 substrate 재사용:** `agent_config_revisions`+`activity_log`+`secret_access_events`로 install/update/approval 감사·버전을 확장.

**엔터티 정합 (스펙→코드).** issue/project/run/cost/skill/plugin **척추는 이미 풍부히 존재**(Company·Project·`Issue`·`AgentProfile`=agents·`Run`=heartbeat_runs·Capability 일부=company_skills/plugins). **협업+proof 레이어는 전부 net-new:** `WorkOwner·PairGroup·PairTurn·ProofBundle·ContextShard·CompactionRun·UsageSnapshot·ArtifactVersion·CapabilitySource/Assignment`. Artifact=`issue_work_products`+`assets`, Annotation=`issue_comments` 매핑이라 **새 코어 테이블은 `ArtifactVersion`만**. ⚠️ WorkOwner는 pair 가정이나 현 이슈는 **엄격 single-owner**(`assigneeAgentId` XOR `assigneeUserId`) — pairing 스키마 발판 없음.

**이슈 상태 모델.** 현 7상태(`backlog·todo·in_progress·in_review·done·blocked·cancelled`) → 목표 10상태. net-new = **`In QA`**(현 QA는 execution-policy review/approval 스테이지일 뿐)·**`Blocked`→Blocked by User/System 분리**(현재 1상태+읽기시점 파생)·**`Archived`**(현재 `hiddenAt` 타임스탬프). Draft/Ready=backlog/todo 리네임. ⚠️ **전이 무가드**(`assertTransition`은 타깃 문자열만 검증) — 실 워크플로는 별개의 execution-policy 스테이지 머신. "Blocked by User 레일"은 blocked-inbox attention(`issues.ts`)으로 **부분 선구현**(파생값·미저장).

**승인/거버넌스 (DecisionRequest).** 재사용 강함: `approvals`(제네릭 type+payload, **board가 결정**(`assertBoard`), 승인 시 agent wakeup, 이슈 링크)=`DecisionRequest`의 본거지. + `issue_execution_decisions`(이슈 내 reviewer/approver 스테이지 원장, 회사 승인 큐와 별개). 스펙 4종 중 **blocked-issue 승인만 기존**(`request_board_approval`); **release sign-off·capability 설치 승인·plan-conflict = net-new type**(특히 skill/plugin 설치엔 현재 승인 게이트 없음). 현 회사 승인은 board 전용 결정(`assertBoard`; reviewer/approver 위임은 `issue_execution_decisions`에 별도) — 여기에 **위임 승인 모드**(D18: 상시 위임 시 `assertBoard` 경로에서 정책 확인 후 기획자 Agent 자동 승인)를 net-new로 얹는다.

**컨텍스트 조립/compaction.** 재사용 donor: `/issues/:id/heartbeat-context`(12-way 번들, 이미 full history 아닌 *요약 문서* 로드) + per-run `contextSnapshot`. ⚠️ **"compaction" 네이밍 함정** — 현존 `session-compaction`은 **CLI 세션 로테이션**(maxSessionRuns/RawInputTokens/AgeHours)이고 토큰-윈도 관리는 **어댑터(claude/codex)에 위임**. 스펙의 4-plane(Charter/Working/Episodic/Capability Index)·`ContextShard`/`CompactionRun`·index+pointer 우선·on-demand는 **전부 net-new**(혼동 금지).

**스케줄러/heartbeat/liveness — 최강 재사용.** wakeup→queue→우선순위 claim→execute 완비: `queueIssueAssignmentWakeup`→`agent_wakeup_requests`→`startNextQueuedRunForAgent`(per-agent lock·`maxConcurrentRuns` 슬롯·dependency/priority/FIFO 정렬·coalescing/idempotency)→`executeRun`. liveness=주기 스케줄러(`index.ts`)가 orphan reap·retry 승급·stranded 재조정·silent-run watchdog 실행. 스펙의 "planner는 멈추지 않고 에이전트가 계속 행동"이 **구조적으로 충족**. evolve from `heartbeat.ts`+`index.ts`.

**업데이트(staged) — 부분 구현.** 플러그인: `plugin-lifecycle.ts` `upgrade()`가 **manifest capability-diff + `upgrade_pending` 승인 게이트** 보유(스펙에 가장 근접). 스킬: `company-skills.ts`가 핀(40자 SHA)+"check updates"는 있으나 `installUpdate`는 **diff/승인 없이 즉시 적용**. net-new = license diff·smoke·breaking-risk·company/agent scope + 스킬 staged-승인 retrofit. evolve from `plugin-lifecycle.ts:639`.

**감사/관측 — substrate 견고(DB 영속).** `activity_log`(`logActivity`: redact+live event+plugin 이벤트버스)·`secret_access_events`가 모든 기록을 DB에 영속한다. 엄격 불변성(UPDATE/DELETE 차단 트리거·권한)은 **선택적 후속 하드닝**(V1 차단 요소 아님). 커버리지=call-site opt-in이라 capability-install/done 판정 로깅만 명시 배선하면 됨. 메트릭/알림 없음(P4).

**멀티컴퍼니 격리 — 강건(재사용).** 모든 테이블 `companyId` FK + `assertCompanyAccess` + `authorization.ts` `deny_company_boundary`. **RBAC는 의도적으로 얇음(P4):** 8개 `PERMISSION_KEYS` + 역할(owner/admin/operator/viewer/member+instance_admin) + grant/scope/manager-chain 엔진(`authorizationService.decide`). board local은 allow-all(단일 테넌트 실용), 에이전트는 default-deny. advanced RBAC=P4. evolve from `authorization.ts`+`PERMISSION_KEYS`.

## 과금 보안 로드맵과 수용 기준

과금 모델은 self-serve 기준으로 **구독형 단일 상품 구조**를 권장한다. 사용자에게는 seat 또는 workspace 기반 구독만 노출하고, 내부적으로는 provider 비용, rate limits, shared credits, burn rate를 별도로 추적해 플랫폼 allowance로 번역한다. Anthropic와 OpenAI 모두 usage, credits, rate limits, spend alerts, usage reports를 공식 표면으로 제공하지만, 이 구조는 공급자별 모델이 다르고 권한도 조직 수준에 묶여 있다. 따라서 최종 사용자 가격 정책은 구독으로 단순화하되, 운영자 화면에서는 provider-level 사용량과 limit 신호를 정밀하게 볼 수 있게 하는 이중 구조가 가장 실용적이다. 다시 말해, **가격은 구독형으로 단순화하고, 운영은 usage-aware로 정교화**해야 한다.

보안 모델의 기본 원칙은 “확장은 열고, 신뢰는 줄일 수 있게”다. Anthropic는 trusted MCP servers만 쓰라고 명시하고, Open Design의 MCP도 read-only이지만 등록한 클라이언트는 로컬 프로젝트 읽기 권한을 얻게 되므로 신뢰 가능한 클라이언트만 등록하라고 경고한다. Paperclip 역시 현재 plugin UI가 trusted same-origin code라고 경고한다. 따라서 본 제품은 capability마다 `trusted`, `reviewed`, `unreviewed` trust tier를 두고, `unreviewed`는 auto-invoke 금지, 설치 후 기본 disabled, 별도 승인 필요, UI bridge 격리, secret 접근 금지 정책을 가져야 한다. secret material은 capability manifest에 넣지 않고 company-scoped secret store에만 보관해야 하며, 설치·업데이트·권한 승인은 모두 audit log에 남겨야 한다.

구현 로드맵은 네 단계가 적절하다. 첫 번째 단계는 Project bootstrap, Planner, Kanban, Issue Workspace, single-owner execution, proof bundle, basic Usage Center다. 두 번째 단계는 PairGroup, round diff, stop policy, blocked decision lane, QA / QC gate, design dashboard 기본형이다. 세 번째 단계는 Open Design bridge, capability registry, scope assignment, user-triggered update, context compaction, visibility control이다. 네 번째 단계는 enterprise hardening으로, adapter smoke tests, provider failure classification, alerting, backup/export, metrics, advanced RBAC를 정리한다. 이 순서는 Paperclip의 control plane을 먼저 활용하고, issueflow 철학으로 운영 규율을 입힌 뒤, Open Design과 외부 능력을 확장하는 방식이라 리스크가 가장 낮다.

MVP 수용 기준은 다음과 같이 고정하는 것이 좋다.

- 신규 프로젝트와 기존 프로젝트 모두 bootstrap이 가능해야 한다. 전자는 plan/anchor/backlog/proof를 새로 만들고, 후자는 repo와 tracker를 스캔해 current state와 backlog를 생성해야 한다.
- 사용자가 자연어로 요청하면 Planner Agent가 ongoing process와 무관하게 draft 또는 ready issue를 생성할 수 있어야 한다.
- WorkOwner는 `SingleAgent`와 `PairGroup` 두 종류를 지원해야 하며, PairGroup은 최대 10라운드, stop reason, round별 diff, 비용 합계를 남겨야 한다.
- UI UX Agent는 Open Design 기반 preview를 생성할 수 있어야 하고, Design Dashboard에서 current / deprecated 화면을 구분해 관리해야 한다.
- Developer Agent는 이슈 단위 vertical slice를 기본으로 하고, 독립 lane은 병렬 worktree 방식으로 dispatch할 수 있어야 한다.
- QA / QC Auditor는 proof bundle 없이는 done 처리할 수 없어야 하며, qualitative 품질 판정도 남길 수 있어야 한다.
- Capability Registry는 외부 skill/plugin/MCP/design system을 company 또는 특정 agent scope로 할당할 수 있어야 하고, user-triggered update와 visibility 상태를 지원해야 한다.
- Usage Center는 provider별로 `Exact`, `Synced`, `Estimated` 중 하나의 정확도 배지와 함께 최근 사용량·잔여량·경고 상태를 보여줘야 한다.
- `PLAN_ANCHOR`, `CURRENT_STATE`, capability index는 compaction 정책과 on-demand loading 정책을 따라 context overflow 없이 유지되어야 한다.
- 모든 mutating action, 승인, provider sync, capability install/update, done 판정은 audit trail로 조회 가능해야 한다.

이 기준을 만족하면, 이 제품은 더 이상 “에이전트가 많은 대시보드”가 아니라 **프로젝트를 실제로 굴리는 사용자 중심 멀티에이전트 운영 플랫폼**으로 기능할 수 있다. 그때 비로소 Paperclip의 control plane, issueflow의 이슈 철학, Open Design의 디자인 엔진, 그리고 사용자가 요청한 인간 중심 회사 모델이 하나의 일관된 제품으로 묶인다.
